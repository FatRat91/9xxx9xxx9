# 1. GCAS認証・権限モデル

## 必須

### ① GCASアカウント → IAM Identity Center → AWSアカウント → アクセス許可セット の流れ

**重要度**: ★★★★★
**必要なスキル・知識**

* GCASアカウントで入ること
* IAM Identity Center を使って AWS アカウントへ入ること
* アクセス許可セットで権限が決まること
* ログイン後は IAM ロールが使われること

**何に使う / どこで役立つ**

* 初回ログイン
* 権限不足時の切り分け
* 「なぜIAM Userを作らないのか」の理解

**根拠**
GCASのAWS利用概要では、**GCASアカウントと連携した AWS IAM Identity Center** により各AWSアカウントへの認証・アクセスが制御されるとされています。AWS公式でも、IAM Identity Center は **ユーザーにアカウントアクセス権を割り当て、必要な権限を自動で構成する** 仕組みです。 ([AWS ドキュメント][2])

---

### ② 管理者 / メンバーの違い

**重要度**: ★★★★★
**必要なスキル・知識**

* 管理者は強い権限を持つ
* メンバーは最初から何でもできるわけではない
* 必要な作業用ロールは追加で作る前提

**何に使う / どこで役立つ**

* 自分が何を依頼すべきかの判断
* 作業開始前の役割確認
* 権限申請時の会話

**根拠**
GCASのAWS利用概要では、CSPユーザー登録時に **管理者 / メンバー** を選択し、メンバーは **スイッチロール以外の特別な権限が付与されない** ため、必要な IAM ロールを管理者が作成するとしています。

---

### ③ IAM User を前提にしないこと

**重要度**: ★★★★★
**必要なスキル・知識**

* IAM User はGCASでは原則使わない
* 恒久アクセスキー前提で考えない
* 人もシステムもロール中心で考える

**何に使う / どこで役立つ**

* 認証方式の誤解防止
* セキュリティレビュー
* IaCや運用設計の前提整理

**根拠**
GCASのユーザー管理方法では、シングルサインオン展開後は **IAMユーザー作成をシステム的に制御し、利用を原則禁止** としています。AWS公式でも、ロール利用により **一時的な認証情報** を使う形が基本です。 ([AWS ドキュメント][3])

---

## 推奨

### ④ アクセス許可セットと IAM ロールの対応関係

**重要度**: ★★★★☆
**必要なスキル・知識**

* Permission Set が IAM ロールに対応すること
* ログイン後にどのロールが使われるか
* 権限の入口は Permission Set であること

**何に使う / どこで役立つ**

* ログイン後の権限確認
* 権限不足時の調査
* SSOまわりの会話理解

**根拠**
AWS公式では、IAM Identity Center の Permission Set は、各アカウント側で **IAM ロールとポリシー** を構成する仕組みです。GCASのユーザー管理方法でも、アクセス許可セットに応じて各AWSアカウントに IAM ロールが作成される構成です。 ([AWS ドキュメント][4])

# 2. IAMポリシー読解

## 必須

### ① Action / Resource / Condition を読める

**重要度**: ★★★★★
**必要なスキル・知識**

* `Action`
* `Resource`
* `Condition`
* `Effect`（Allow / Deny）

**何に使う / どこで役立つ**

* ポリシーの読み取り
* 権限不足の調査
* 危ない権限の発見

**根拠**
GCASのユーザー管理方法では、IAMポリシーは **アクション（操作）、操作対象リソース、条件** を制御する JSON 形式の文書と説明されています。AWS公式でも、IAM JSON policy elements の中核として **Action / Resource / Condition** が定義されています。 ([AWS ドキュメント][5])

---

### ② 最小権限で考える癖

**重要度**: ★★★★★
**必要なスキル・知識**

* とりあえず `*` を付けない
* 必要な操作だけ許可する
* 参照専用でも専用ロールを切る

**何に使う / どこで役立つ**

* ロール設計
* CDKレビュー
* セキュリティレビュー

**根拠**
GCASのユーザー管理方法では、**リソース参照のみの場合も専用のIAMロールを作成**し、権限最小化を念頭に設計するよう示しています。AWS公式でも IAM ポリシーは必要なアクセスに限定して設計する前提です。 ([AWS ドキュメント][5])

---

## 推奨

### ③ AWS管理ポリシーとカスタマー管理ポリシーの違い

**重要度**: ★★★★☆
**必要なスキル・知識**

* AWS管理ポリシー
* カスタマー管理ポリシー
* どちらを使うと変更追従しやすいか

**何に使う / どこで役立つ**

* 既存権限の流用判断
* ポリシー設計
* PRレビュー

**根拠**
GCASのユーザー管理方法では、IAMポリシーは **自分で作成するカスタマー管理ポリシー** と **AWSが管理するAWS管理ポリシー** に分かれ、AWS管理ポリシーはサービス拡張に合わせて更新されるとしています。AWS公式でも、管理ポリシーの使い分けが案内されています。 ([AWS ドキュメント][6])

---

### ④ ポリシー差分レビュー

**重要度**: ★★★★☆
**必要なスキル・知識**

* before / after 比較
* `*` の増加に気づく
* `Condition` の削除に気づく
* 本番系だけ権限が広がっていないか確認する

**何に使う / どこで役立つ**

* PRレビュー
* 本番変更レビュー
* 監査対応

**根拠**
これはGCAS文書の明文化というより、**IaC原則・最小権限原則**から直接出る実務上の必須寄り推奨です。GCASはインフラ変更を IaC 原則とし、管理コンソール参照のみを原則にしています。

---

# 3. IAMロール / スイッチロール / 信頼ポリシー

## 必須

### ① IAMロールと一時認証情報

**重要度**: ★★★★★
**必要なスキル・知識**

* IAMロールは永続認証情報を持たない
* 利用時に一時認証情報が発行される
* AssumeRole の意味

**何に使う / どこで役立つ**

* 実行ロール理解
* スイッチロール理解
* 外部連携理解

**根拠**
GCASのユーザー管理方法では、IAMロールは **永続的な認証情報を持たず、一時的な認証情報を発行**するとされています。AWS公式でも、AssumeRole は **一時的なセキュリティ認証情報を発行する** API です。 ([AWS ドキュメント][3])

---

### ② 信頼ポリシーを読める

**重要度**: ★★★★★
**必要なスキル・知識**

* `Principal`
* `Action: sts:AssumeRole`
* `Condition`
* どの主体がそのロールを使えるか

**何に使う / どこで役立つ**

* スイッチロール設定
* クロスアカウント連携
* 誤った権限公開の防止

**根拠**
GCASのユーザー管理方法では、利用システム側で作成したIAMロールに対して、**SwitchOnlyRole からスイッチロールできるよう信頼ポリシー設定が必要**であり、**IdCユーザー名を個別に指定したアクセス制御を強く推奨**しています。AWS公式でも信頼ポリシーでは Principal や Condition による制御を行います。 ([AWS ドキュメント][7])

---

### ③ スイッチロール前提の作業イメージ

**重要度**: ★★★★★
**必要なスキル・知識**

* メンバーはまず強権限を持たない
* 作業用ロールへ切り替えて作業する
* そのための信頼ポリシーが必要

**何に使う / どこで役立つ**

* 日常保守
* 一時的な高権限作業
* 本番権限制御

**根拠**
GCASのAWS利用概要では、管理者以外のユーザーには **スイッチロール以外の特別な権限を付与しない** としており、必要なIAMロールと信頼ポリシーを管理者が用意する前提です。

---

## 推奨

### ④ クロスアカウント AssumeRole の安全な考え方

**重要度**: ★★★★☆
**必要なスキル・知識**

* 別アカウントのロール利用
* Principal を相手アカウントに絞る
* Condition でさらに縛る

**何に使う / どこで役立つ**

* 外部システム連携
* 非同期連携
* ファイル連携

**根拠**
リファレンスアーキテクチャでは、外部システムが別AWSアカウントにある場合、信頼ポリシーで **相手アカウントIDを Principal に指定し、さらに Condition に一意な値を指定**することを推奨しています。

# 4. GCASガバナンスの土台

## 必須

### ① Control Tower 管理下のアカウント分離

**重要度**: ★★★★★
**必要なスキル・知識**

* AWSアカウントは Control Tower 管理
* 本番用アカウント / 検証用アカウントを分ける
* 開発環境は原則ない

**何に使う / どこで役立つ**

* 環境構成の理解
* 権限設計
* CI/CD設計
* 課金分離の理解

**根拠**
GCASのAWS利用概要では、AWSアカウントは **デジタル庁の Control Tower により管理**され、**本番用アカウントと検証用アカウントを分ける**こと、**開発環境は原則提供しない**ことが明記されています。 ([guide.gcas.cloud.go.jp][8])

---

### ② 既にポリシーが入っていて、利用組織は変更できない

**重要度**: ★★★★★
**必要なスキル・知識**

* 払い出し時点でガバナンス・セキュリティ設定が入っている
* 利用組織側で全部自由には変えられない
* まずGCASの制約を確認する

**何に使う / どこで役立つ**

* できない操作の切り分け
* 例外相談の要否判断
* 設計時の前提整理

**根拠**
GCASガイドサマリーでは、払い出されるアカウントには **ガバナンス・セキュリティに係るポリシーが設定済み** で、**利用組織はこの設定を変更できない** とされています。 ([guide.gcas.cloud.go.jp][9])

---

### ③ 必須適用テンプレート

**重要度**: ★★★★★
**必要なスキル・知識**

* 自動適用テンプレート
* 必須適用テンプレート
* サンプルテンプレート
* “必須” と “任意” の違い

**何に使う / どこで役立つ**

* 初期設定
* セキュリティベースライン理解
* IaC設計

**根拠**
GCASガイドサマリーでは、**すべての払い出しアカウントに必須適用テンプレートを適用する必要**があると整理されています。地方公共団体向け文書でも、GCASテンプレートは **自動適用 / 必須適用 / サンプル** の3種類で構成されると定義されています。 ([guide.gcas.cloud.go.jp][9])

---

### ④ IaC原則、コンソールは参照中心

**重要度**: ★★★★★
**必要なスキル・知識**

* インフラ構築・変更は IaC が原則
* 管理コンソールは参照中心
* 手作業変更を前提にしない

**何に使う / どこで役立つ**

* 日常運用
* CDK運用
* 監査対応
* 障害時の変更判断

**根拠**
GCASガイドサマリーでは、**インフラの構築・変更はIaC原則**、**管理コンソールは参照時のみ原則** とされています。 ([guide.gcas.cloud.go.jp][9])

---

## 推奨

### ⑤ SCP / 自動適用テンプレート / 必須適用テンプレート の役割分担

**重要度**: ★★★★☆
**必要なスキル・知識**

**根拠**
GCASのAWS利用概要では、ガバナンス・セキュリティは **SCP と自動適用テンプレート、必須適用テンプレート** によって担保されるとされています。

# 最後に一言

**足切り回避だけなら、IAMの細かい暗記より、GCASの権限モデルと統制モデルを誤解しないことの方が重要です。**

最優先で固めるなら、この順です。

1. **GCASアカウント / IdC / アクセス許可セット / ロール**
2. **IAM User ではなくロール中心**
3. **Action / Resource / Condition**
4. **スイッチロールと信頼ポリシー**
5. **Control Tower / 必須適用テンプレート / IaC原則**
6. **“権限不足” と “統制制約” の切り分け**
