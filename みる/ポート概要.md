## 結論 🎯

**今回の案件（ガバメントクラウド + Windows リプレイス寄り）で、SG で最初に意識する頻出ポートは次です。**

### 最優先で覚えるもの

* **443/TCP** … 本命。Web、API、認証、外部連携の中心
* **80/TCP** … 主に **HTTPS へのリダイレクト用**
* **53/TCP, UDP** … DNS
* **DB系** … **3306 / 5432 / 1433 / 1521** のどれか
* **22/TCP / 3389/TCP** … SSH / RDP だが、**常時開放は極力避ける**
* **445/TCP / 2049/TCP** … ファイル共有系を使うときだけ
* **AD連携系** … **88 / 389 / 464 / 636 / 3268 / 3269 / 5985 / 9389 / 49152-65535**
* **メール送信系** … **587 / 465**（必要時）。**25 は避け寄り**

---

## まず大前提（事実）📌

ガバメントクラウド側の考え方として、**通信で本当に管理すべきなのは「自分で明示的に開けたポート」** で、外部公開は基本的に **HTTPS中心** です。GCAS/デジタル庁資料でも、**HTTP→HTTPS リダイレクト**、**HTTPS必須**、**許可した通信のみ通す** ことが明示されています。  

また、共通セキュリティ資料では **HTTP や FTP のような平文通信は使わず、閉域網であっても HTTPS や SFTP を使う** 方針です。

さらに、ガバメントクラウドは **OS にログインしない運用** を強く志向しており、マネージドサービスやコンテナ、サーバレスの活用で **踏み台や OS ログイン前提の運用を減らす** 方向です。 

---

## ポート一覧（今回の案件向け）🧭

# 1. Web / API / フロント公開

### **443/TCP**

* **用途**: HTTPS、Web公開、API公開、認証連携、外部システム連携
* **今回の重要度**: **最重要**
* **理由**:

  * リファレンスアーキテクチャでは、CloudFront・API Gateway・WAF を使う前提が多く、**HTTPS を強制** する設計です。 
  * Replatform資料でも、ALB は **HTTP/HTTPS**、NLB は **TCP/TLS/UDP** とされ、L7公開は HTTPS 中心です。
* **SGの考え方**:

  * インターネット向け公開は、まず **443 だけ** を第一候補にする
  * 送信元は `0.0.0.0/0` になり得るが、**置き場所は CloudFront / ALB / API Gateway 側** に寄せる

---

### **80/TCP**

* **用途**: HTTP
* **今回の重要度**: **高いが補助**
* **理由**:

  * そのまま平文通信で使うより、**HTTPS へリダイレクトするための受け口** として使うのが基本です。GCAS資料でも **HTTP→HTTPS リダイレクト** が明示されています。 
* **SGの考え方**:

  * インターネット公開で 80 を開けるなら、**リダイレクト専用** と考える
  * アプリ本体へ 80 を直接当て続ける設計は避け寄り

---

# 2. DNS / 名前解決

### **53/TCP, UDP**

* **用途**: DNS
* **今回の重要度**: **高い**
* **理由**:

  * リファレンスアーキテクチャでは **Route 53 による DNS 機能** が前提です。
  * Windows/AD/FSx を絡めると、**DNS はほぼ必須級** になります。FSx for Windows の AD 参加要件でも **53/TCP,UDP** が必要です。([AWS ドキュメント][1])
* **SGの考え方**:

  * **公開DNS** 用というより、**AD DNS / 名前解決** 用の内部通信として意識することが多い
  * 送信元を **アプリSG / FSx / 管理系SG** に限定する

---

# 3. DB 接続

### **3306/TCP**

* **用途**: MySQL / MariaDB / Aurora MySQL
* **今回の重要度**: **高い**
* **理由**:

  * GCASのリファレンスでは、OSSや業務APIの保管先として **Aurora**、また MySQL/PostgreSQL 系の利用が繰り返し出てきます。
  * AWS公式でも RDS の一般的ポートとして **MySQL / MariaDB = 3306** が示されています。([AWS ドキュメント][2])
* **SGの考え方**:

  * **インターネットから開けない**
  * **App SG → DB SG** のみ許可

---

### **5432/TCP**

* **用途**: PostgreSQL / Aurora PostgreSQL
* **今回の重要度**: **高い**
* **理由**:

  * GCASリファレンスで PostgreSQL 対応OSS → Aurora 利用が示されています。
  * AWS公式の一般的ポートは **PostgreSQL = 5432** です。([AWS ドキュメント][2])
* **SGの考え方**:

  * これも **App SG からだけ**
  * 運用端末や踏み台から直で開ける設計は最小限にする

---

### **1433/TCP**

* **用途**: Microsoft SQL Server
* **今回の重要度**: **中〜高**
* **理由**:

  * Windows リプレイス案件では、現行DBが SQL Server の可能性が普通にあります
  * AWS公式の一般的ポートは **SQL Server = 1433** です。([AWS ドキュメント][2])
* **SGの考え方**:

  * SQL Server 採用時のみ
  * **アプリSG限定**、外部公開禁止

---

### **1521/TCP**

* **用途**: Oracle
* **今回の重要度**: **中**
* **理由**:

  * Oracle 現行踏襲案件だと候補になります
  * AWS公式の一般的ポートは **Oracle = 1521** です。([AWS ドキュメント][2])
* **SGの考え方**:

  * 使うときだけ
  * 原則 **閉域・内部限定**

---

## ここは事実 / 推測の区別 👇

> **事実**
> GCAS資料では Aurora や各種マネージドDB利用が繰り返し出ます。
>
> **推測・実務一般**
> 今回の案件で実際に開ける DB ポートは **3306 / 5432 / 1433 / 1521 の全部ではなく、採用DBの1つか2つ** です。

---

# 4. 管理アクセス（できれば減らす）

### **22/TCP**

* **用途**: SSH、または SFTP
* **今回の重要度**: **中**
* **理由**:

  * Linux 管理や一部ファイル転送ではまだ出ます
  * ただし AWS Systems Manager Session Manager は、**インバウンドポートを開けずに管理できる** と公式に明記しています。([AWS ドキュメント][3])
  * Transfer Family の SFTP では、公開エンドポイントは **22**、VPC 配置時は **22 / 2222 / 2223 / 22000** も使えます。([AWS ドキュメント][4])
* **SGの考え方**:

  * 管理用に開けるなら **特定IP/CIDRのみに限定**
  * できれば **22 を開けずに Session Manager** に寄せる

---

### **3389/TCP**

* **用途**: Windows の RDP
* **今回の重要度**: **中**
* **理由**:

  * Windows リプレイスだと、どうしても最初は意識します
  * ただし AWS 公式でも **22/3389 は限定IPだけに許可** を強く推奨しています。([AWS ドキュメント][5])
  * しかも Session Manager は **開放不要** の管理手段です。([AWS ドキュメント][3])
* **SGの考え方**:

  * `0.0.0.0/0` はかなり避けたい
  * **踏み台前提より、SSM/Fleet Manager/私設トンネル** を優先

---

## 実務メモ 📝

> **事実**
> ガバメントクラウドは OSログイン前提を減らす方向です。 
>
> **推測**
> なので、**22/3389 は「頻出だけど、理想は閉じたいポート」** と覚えるのが実務寄りです。

---

# 5. ファイル共有 / ストレージ

### **2049/TCP**

* **用途**: NFS（EFS）
* **今回の重要度**: **中**
* **理由**:

  * Replatform資料では、ファイルストレージが必要なら **EFS / FSx for Windows** を検討するとあります。
  * EFS の公式SGルールは **NFS 2049** です。([AWS ドキュメント][6])
* **SGの考え方**:

  * **EC2 SG ↔ EFS SG** の間だけ
  * インターネット公開しない

---

### **445/TCP**

* **用途**: SMB / Windows ファイル共有 / FSx for Windows
* **今回の重要度**: **中**
* **理由**:

  * Windows リプレイスでファイルサーバ系が残るなら候補
  * FSx for Windows の AD 参加要件にも **445** が含まれます。([AWS ドキュメント][1])
* **SGの考え方**:

  * **Windows サーバ群 / FSx / ドメイン参加対象** に限定
  * かなり閉じた範囲で使う

---

# 6. Windows / Active Directory 連携で必要になるポート

これは **AD や FSx for Windows を使うときだけ** です。
ただし、Windows リプレイスではかなり出やすいです。

### よく出るもの

* **53/TCP, UDP** … DNS
* **88/TCP, UDP** … Kerberos
* **464/TCP, UDP** … パスワード変更
* **389/TCP, UDP** … LDAP
* **636/TCP** … LDAPS
* **123/UDP** … NTP
* **135/TCP** … RPC Endpoint Mapper
* **3268/TCP** … Global Catalog
* **3269/TCP** … Global Catalog over SSL
* **5985/TCP** … WinRM
* **9389/TCP** … AD DS Web Services / PowerShell
* **49152-65535/TCP** … 動的RPCポート
* **445/TCP** … SMB

これらは AWS 公式の FSx for Windows + self-managed AD 要件でまとまって示されています。([AWS ドキュメント][1])

### SGの考え方

* **全部を広く開けるものではない**
* **AD / FSx / 管理対象サーバ間** のみ
* **NACL や OS ファイアウォール** では双方向要件も意識する
  （AWS公式でも、SGは通信開始方向だけでよい一方、Windows FW や NACL は両方向で考える必要があるとされています） ([AWS ドキュメント][1])

---

# 7. メール送信

### **587 / 465 / 2587 / 2465**（必要時）

* **用途**: SES SMTP
* **今回の重要度**: **中**
* **理由**:

  * GCAS資料でも **Amazon SES SMTP インターフェースを用いたメール送信** はユースケース例として出ます。
  * AWS公式では STARTTLS は **25 / 587 / 2587**、TLS Wrapper は **465 / 2465** です。([AWS ドキュメント][7])
* **SGの考え方**:

  * 基本は **送信（outbound）側**
  * 受信SMTPサーバを自前で立てないなら、インバウンドは普通不要

---

### **25/TCP**

* **用途**: SMTP
* **今回の重要度**: **低め**
* **理由**:

  * EC2 から SES へ送る場合、AWS は **25番制限** に触れています。代わりに **465 / 587** が推奨寄りです。([AWS ドキュメント][8])
* **SGの考え方**:

  * 使わなくて済むなら使わない
  * **587 / 465 優先** で考える

---

# 8. ファイル転送

### **22/TCP**

* **用途**: SFTP
* **理由**:

  * 共通セキュリティでは VPC外通信でも **SFTP** を使う方針です。
  * Transfer Family の SFTP は標準で **22** です。([AWS ドキュメント][4])

### **21/TCP**

* **用途**: FTP / FTPS 周辺
* **理由**:

  * GCAS資料でも Transfer Family が出てきます。
  * AWS公式では FTP/FTPS 構成の注意点として **NLB のヘルスチェックに 21 を使う** 記述があります。([AWS ドキュメント][4])

---

## 注意 ⚠️

> **事実**
> GCAS資料は「HTTPS / SFTP寄り」で、平文の HTTP / FTP は避ける方向です。
>
> **推測**
> したがって **FTP(21) は“使うことはあるが本命ではない”**、**SFTP(22) の方がまだ出やすい** と見てよいです。

# 10. SG設計の要点だけ最後にまとめると ✅

* **インターネット公開は基本 443**
* **80 はリダイレクト用**
* **DBポートは App SG からのみ**
* **22/3389 は原則しぼる。可能なら閉じる**
* **AD系ポートは“Windows採用時だけ”**
* **ファイル系は 445 / 2049 を必要時だけ**
* **メール送信は 587 / 465 を優先**
* **「開ける理由が説明できないポートは開けない」**
  これがガバメントクラウド寄りの考え方です。 

必要なら次に、**「このポート一覧をそのまま CDK の `SecurityGroup` ルール例に落とした版」** を、
**ALB / EC2(App) / RDS / FSx / 管理用** のSG構成に分けて出します。
