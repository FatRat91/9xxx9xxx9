## 結論 🎯

**90時間（70日、平日1h・土日2h）で目指すべきなのは、14項目を“深く全部”ではなく、**
**「足切りされないための必須ライン」** と **「入ってからかなり楽になる推奨ライン」** を分けて押さえることです。

**事実**として、GCASガイドは今も更新が続いており、2026年には **GCAS Connect 利用ガイド** や **運用モダン化実践ガイド** が追加されています。つまり、案件で見られるのは古い静的知識ではなく、**GCASの流れ・役割分担・モダン化の考え方を前提にした実務対応力**です。([GCASガイド][1])

**推論**として、この90時間では

* **前半6項目を厚く**
* **次の5項目を“手を動かせる最低限”**
* **最後の3項目を“浅くても会話できる”水準**
  に切るのが、いちばん実務的です。

# 1. GCAS特有のルール・流れ

## 必須

* **GCASガイドのどこに何があるか**を言える
* **利用開始までの流れ**を言える
  例：現行確認 → 移行計画 → 予算要求・調達 → 契約 → 環境払い出し → 初期設定 → 移行
* **環境・ユーザー・ネットワーク・責任分界・調達・利用の流れ** がGCAS入門の主軸だと理解している
* **必須適用テンプレート**の存在を知っている

## 推奨

* 定例で単語が出たら、**GCASガイドの該当ページを即引ける**
* **GCAS Connect** や最近追加された運用系ドキュメントの存在まで把握している

## 理由

GCASサマリー自体が、最初に見る観点として **環境とユーザー、ネットワーク、責任分界点、調達、利用の流れ、ユーザー登録** を並べています。また地方公共団体向け手順書でも、移行は単なる構築ではなく、調達・契約・回線・環境払い出しまで含む流れです。  ([GCASガイド][1])

# 2. リプレイス

## 必須

* **R1（Replatform）と R2（Rebuild）の違い**を説明できる
* **APIベース / ステートレス / マネージド活用 / 運用のコード化 / サービスレベル計測** がGCASのモダン化定義だと理解している
* **Lambda → Fargate → EC2** の大まかな判断軸を持っている
* **バッチ前提からイベントドリブン寄りへ** という方向性を理解している

## 推奨

* 現行オンプレ構成を見て
  **「この部分はそのまま」「この部分はマネージドへ」「この部分は要改修」**
  を雑にでも分けられる
* **セッション外出し / ファイル外出し / ジョブ分離** を移行論点として拾える

## 理由

GCASのモダン化定義では、2025年時点のモダン化として **APIベース、ステートレス、マネージドサービス活用、運用のコード化・自動化、サービスレベルの定義・計測** を定義し、**R1はマネージド活用が原則、R2は1〜5全体** が対象です。運用ノウハウでも、**Lambdaで足りればLambda、足りなければFargate、特殊要件ならEC2/ECS on EC2** と整理されています。

# 3. IAMガバナンス

## 必須

* **GCASアカウント → IAM Identity Center → アクセス許可セット → IAMロール** の流れを理解している
* **IAMユーザーを増やす発想ではない**ことを理解している
* **管理者 / 管理者以外（SwitchOnlyRole）** の違いを説明できる
* **本番管理者は3名まで推奨**を知っている

## 推奨

* IAMポリシーJSONを見て **Action / Resource / Condition** を読める
* **スイッチロール前提の運用**を説明できる
* **CDK bootstrap まわりで追加権限が必要になるケース**を軽く理解している

## 理由

GCASのAWS環境では、GCASアカウントによるSSOが前提で、利用組織で別ユーザーを作る必要はないとされています。さらに IAMロール前提で、管理者以外には **SwitchOnlyRole**、本番管理者は **3名まで推奨**です。

# 4. NW（ネットワーク）

## 必須

* **VPC / Subnet / Route / Security Group / NAT / ALB** の役割を説明できる
* **本番と検証を分ける理由**を説明できる
* **他環境とは許可しない限りつながらない**前提を知っている
* **回線・接続方式が別工程で重い論点**だと理解している

## 推奨

* 簡単なNW図を見て **通信経路を口頭で説明**できる
* **GCAS Connect** の存在を知っておく
* **踏み台前提ではなく閉じる方向**で考えられる

## 理由

GCASサマリーでは、環境は分離され、**許可しない限り他環境とネットワーク接続されない** とされています。AWS利用概要でも **本番用アカウントと検証用アカウントを分ける** と明記され、移行手順書でも **ネットワーク接続方法** は早期着手項目です。   ([GCASガイド][2])

# 5. 調達・契約・責任分界

## 必須

* **単独利用方式 / 共同利用方式** の違いを説明できる
* **運用管理補助者 / ASP / 回線運用管理補助者** のざっくりした役割を分けられる
* **予算要求・RFI/RFP・契約締結** が技術外の大論点だと理解している
* **利用料見積りの精緻化** が必要だと知っている

## 推奨

* 会議で事業者名が出たときに
  **「その人は何の受託者か」** を推測できる
* 共同利用方式がなぜ推奨されるかを説明できる

## 理由

地方公共団体向け文書では、共同利用方式は **手続き簡素化** と **運用負担軽減** が主なメリットです。移行手順も **移行計画 → 予算要求・調達 → 契約手続き** まで明確に章立てされており、ここを知らないと現場会話で置いていかれやすいです。

# 6. CDK TypeScript（IaC）

## 必須

* **Construct / Stack / props / synth / diff / deploy** を説明できる
* **CDK → CloudFormation** の流れを理解している
* `ec2.Vpc` / `ec2.SecurityGroup` / `iam.Role` / `s3.Bucket` / `lambda.Function` あたりのコードを読める
* **CloudFormationイベントで失敗箇所を追える**

## 推奨

* 小さな1スタックを **自力で書いて diff まで見られる**
* **API Reference を見て props を自力で追える**
* Construct Library の **L2 / L1 の違い** を理解している

## 理由

デジタル庁のIaC資料では、GCASが配布するIaCテンプレートはAWSでは **AWS CDK** 前提で、環境払い出しもIaCで自動化され、全環境に同じ保護機能を強制するガバナンスを実現するとしています。AWS CDK公式でも、Construct はコンストラクタに props を取り、サービス別に `aws-cdk-lib.<service>` で整理されています。 ([AWS ドキュメント][3])

# 7. CI/CD

## 必須

* **インフラCI/CD と アプリCI/CD の違い**を説明できる
* **ソース → ビルド → テスト → 承認 → デプロイ** の流れを言える
* **ベンダーロックインが強いツールは避ける**方針を知っている
* **GCAS指定のツール / テンプレート / IaCファイル優先** を理解している

## 推奨

* Gitで **ブランチ → PR → レビュー → マージ** を日常操作として回せる
* `cdk synth` / `cdk diff` / `cdk deploy` を最低1回は流したことがある

## 理由

GCASのデリバリーパイプライン構築ガイドは、**本書はインフラのデリバリーを対象** とし、アプリCI/CDはシステムごとに最適化、さらに **特定ベンダーだけが使えるツールは避ける** と明記しています。AWS利用概要でも、インフラ部分のCI/CDはGCAS指定ツール・テンプレート・IaC優先です。  ([GCASガイド][4])

# 8. 監視（Observability）

## 必須

* **Monitoring / Observability / Visualization** の違いを説明できる
* **CPUだけ見ればよいわけではない** と理解している
* **KPI / サービスレベル / UXへの影響** を見る発想を持つ
* CloudWatchで **ログ / メトリクス / アラーム** の基本を理解している

## 推奨

* **運用者向け指標** と **サービス改善向け指標** を分けて考えられる
* 1枚でよいので **ダッシュボードの項目案** を作れる

## 理由

デジタル庁の監視記事は、CPU使用率監視だけではなく **UXに影響する指標** を見て、**サービス改善や価値向上**に寄せたモニタリングが重要だと説明しています。GCASでも、定量的計測の活用として **Monitoring / Observability / Visualization** のテンプレートやサンプルを整備する方向です。

# 9. データベース / ストレージ移行

## 必須

* **RDS / Aurora / DocumentDB / S3** のざっくりした使い分けを説明できる
* **S3に置くならオブジェクトキー管理が必要**と理解している
* **セッションや一時状態をサーバ外へ出す**発想を持つ
* **RDB固有機能を同等/代替/廃止で考える**視点を持つ

## 推奨

* **バックアップ / 復旧 / 監視 / 切替** の観点で移行論点を拾える
* **ファイル保管はS3、半構造化はDocumentDB候補** くらいの判断ができる

## 理由

GCASのモダン化定義では **オブジェクトストレージ化、データベース、バックアップ等のマネージド活用** を重視しています。Replatform系の整理でも、**セッションは外部保持**、S3活用、DBのマネージド化が強い論点です。  ([GCASガイド][5])

# 10. SSM

## 必須

* **Session Manager** の利点を説明できる

  * ポート開放不要
  * 踏み台不要
  * SSH鍵不要
* **Patch Manager** が何を自動化するか分かる
* **Automation runbook** の役割を説明できる

## 推奨

* **Session Manager で接続**
* **Run Command / Patch / Automation** の違いを整理
* runbook を見て **何を自動化しているか読める**

## 理由

Session Manager は **IAMで集中制御でき、インバウンドポート・踏み台・SSH鍵管理が不要**です。Patch Manager は **パッチ運用の自動化とコンプライアンス把握**、Automation は **runbook で定型作業を自動化**します。実務でOSやノードに触る場面があるなら、ここは費用対効果が高いです。([AWS ドキュメント][6])

# 11. KMS（データ暗号化）

## 必須

* **AWS managed key / customer managed key** の違いを説明できる
* **自動ローテーションが使える条件**を知っている
* **Key policy が重要**だと理解している
* **S3 / RDS / Secrets Manager と KMS の関係**を説明できる

## 推奨

* **CMK を使うべきケース / AWS managed で足りるケース** を考えられる
* **オンデマンドローテーション** の存在も知っている

## 理由

AWS KMS公式では、自動ローテーションは主に **customer managed の対称暗号KMSキー** が対象で、AWS managed keys は **毎年自動ローテーション** されます。さらに現在は **カスタムの rotation period（90〜2560日）** や **on-demand rotation** もあります。GCAS案件では暗号化そのものより、**どこでどう使うか** を説明できることが重要です。([AWS ドキュメント][7])

# 12. コンテナ

## 必須

* **ECS on Fargate / ECS on EC2 / EC2** の違いを説明できる
* **Lambdaで足りなければFargate優先** を理解している
* **task / service / ALB / logs / health check** の関係を言える

## 推奨

* Dockerfile を見て **何をしているか** 読める
* **コンテナ化可能かどうか** をざっくり判断できる

## 理由

GCASの運用マネージド化ノウハウでは、**Lambdaで要件を満たせればLambda、満たせなければなるべくコンテナ化してFargate、特殊要件ならEC2/ECS on EC2** と整理されています。だから参画前は Kubernetes 深掘りより、**Fargateの基本理解** のほうが効きます。

# 13. FinOps

## 必須

* **契約前の値引きより、稼働後の継続的なコスト削減が効く** ことを理解している
* **サイズ見直し / 停止 / ストレージ選択 / オブジェクト化** が基本だと知っている

## 推奨

* 月次で見る **簡単なコストレビュー観点** を持っている
* 検証環境で **止められるものは止める** 発想がある

## 理由

デジタル庁は、クラウドでは **本番稼働後の運用の工夫による継続的コスト削減** のほうが大きな効果を出せると説明しています。開始前は深掘りよりも、**“運用で削るもの”** という感覚を持つだけで十分価値があります。

# 14. 汎用知識

## 必須

* **Git / PR / レビュー**
* **AWS CLI + SSO（IdC）**
* **HTTP / JSON / REST**
* **CloudFormationイベントの見方**
* **ログを見て一次切り分けする力**

## 推奨

* CLIで **profile切替 / 認証 / 基本操作** ができる
* **CDKエラー → CloudFormationイベント → ログ** の順で追える
* Construct と module の考え方を言える

## 理由

これはGCAS固有ではなく、**入ってから作業者として止まらないための最低限**です。CDK公式も、Construct はアプリの基本部品で、Construct Library はサービス別モジュールに整理されているとしています。IAM Identity Center も現行AWSで継続利用されており、CLI側の `aws configure sso` など旧 namespace が後方互換で残っています。([AWS ドキュメント][3])

# かなり大事な一言

**この90時間で一番危ないのは、CDKやコンテナばかりやって、GCASの流れ・責任分界・IAM/NWを薄くすることです。**

**GCAS特有のルール・流れ → リプレイス → IAMガバナンス → NW → 調達・契約・責任分界 → CDK**
