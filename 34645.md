前提 🧭

ここでの 「APIRefactoring」 は、前回の文脈どおり API Reference として扱います。
また、以下は 前回こちらが列挙した props / API Reference を全部そのまま対象にして、「その1項目を見たときに何を理解すべきか」 を 1行ずつ にしたものです。
CDK の API Reference 自体は、サービス別サブモジュールで整理され、各 Construct ページに Construct Props がまとまっています。

0. まず API Reference 全般の見方 📚

Construct のページ: そのリソースが 何を表すか / 何を暗黙作成するか / どの補助メソッドを持つか を把握する入口。

*Props のページ: new Xxx(this, 'Id', props) に 最初に渡す設計値だけ を確認するページ。

Cfn* のページ: L2 で足りず CloudFormation 生値や未ラップ機能を使う時だけ 見るページ。

モジュール README: 単一クラスより前に、推奨パターン・deprecated・組み合わせ方 を掴むページ。

9. かなり重要な補足 ⚠️
事実

CloudFront の S3 origin は、前回触れた通り S3Origin ではなく S3BucketOrigin / S3StaticWebsiteOrigin を見るのが新しい形 です。

ECS Cluster の旧 containerInsights は、今は containerInsightsV2 も確認対象 です。

Step Functions は definition より definitionBody が推奨です。

EC2 では keyName より keyPair 側へ寄っている箇所が増えています。

推論

実務で読む優先順は、Construct 本体 → Props → README / 例 → 必要時だけ Cfn(L1) が一番速いです。

特にあなたの用途だと、props の意味を暗記するより「その props で何を判断するか」 を覚えた方が、CDK 読解がかなり早く進みます。
