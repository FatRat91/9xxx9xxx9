## 結論 📌

この5項目は、**「オンプレ構成をそのままAWSに置く」のではなく、運用負荷と将来改修コストを下げる方向に寄せる**ための考え方です。
特に GCAS では、モダン化の定義として **①APIベース ②ステートレス ③マネージド活用 ④運用のコード化・自動化 ⑤サービスレベルの定義・計測** を置いており、**R1（Replatform）でも最低限③は原則要求、R2（Rebuild）では①〜⑤全体**が求められます。 ([GCASガイド][1])

---

# 1. APIベース / ステートレス / マネージド活用 / 運用のコード化 / サービスレベル計測

## まず意味を噛み砕くと

### APIベース

システム同士を**直接DB共有やファイル受け渡し中心でつなぐのでなく、APIで疎結合に連携する**考え方です。
GCASでは、API連携にすると **最新データを扱いやすく、エラーハンドリングもしやすく、障害影響を小さくできる** と整理しています。さらに、時間のかかる処理は **APIで受け付けた後に永続化し、後続はイベントドリブンで順次処理**する形を推奨しています。

### ステートレス

**どのサーバで処理しても同じ結果が返る状態**です。
GCASでは、ステートレス化により **障害時の信頼性向上** と **負荷に応じた台数増減のしやすさ**、つまりスケールしやすさとコスト最適化につながるとしています。

### マネージド活用

**監視、ログ、バックアップ、DB、ストレージ、ジョブ実行などを“できるだけAWS側に持たせる”** 考え方です。
R1でもここは原則要求で、Replatform の中核です。 ([GCASガイド][1])

### 運用のコード化・自動化

**手作業でサーバに入って設定・変更するのでなく、IaCや自動ジョブで再現可能にする**ことです。
GCASは、クラウドではサービスイン後も継続的に変えるのが前提で、**本番ログインや手作業依存を減らす**方向をモダン化に含めています。 ([GCASガイド][1])

### サービスレベル計測

**CPU使用率だけでなく、“ユーザーに価値を届けられているか” を KPI や SLO 的に測る**ことです。
GCASでは KPI とサービスレベルを定義し、**ダッシュボード可視化 → 計測 → 振り返り → 改善** を重視しています。デジタル庁の監視記事でも、単なるシステム監視ではなく **UXに効く監視** を重視しています。  

## 案件参画までに最低限わかっておくこと

* APIベースは、**「同期APIだけ」ではなく、重い処理は非同期化まで含む**こと。
* ステートレスは、**セッションや途中状態をアプリサーバに持たない**こと。
* R1でも「とりあえずEC2に移す」だけでは弱く、**マネージド置換を常に考える**こと。
* 監視は、**CPU/メモリ監視だけでは不十分**で、UXや処理時間も見ること。

# 2. Lambda → Fargate → EC2 の判断軸

## GCASの整理はかなり明確です

### Lambda を選びやすい条件

* **Lambdaが対応する言語/ツールで実装できる**
* **1回の実行が15分以内**で終わる
* あるいは **15分以内のサブタスクに分割できる** 

AWS公式でも Lambda のタイムアウト上限は **900秒（15分）**です。([AWS ドキュメント][2])

### Fargate を選びやすい条件

* **15分を超える**
* Lambda非対応の言語/ツールが必要
* ただし **コンテナ化は可能**
* 特殊インスタンスは不要 

GCASは、Lambdaで満たせない場合でも **なるべくコンテナ化し、サーバーレス環境を優先** としています。Fargate はコンテナホスト管理が不要で、イベント駆動起動やロードバランサ連携、オートスケールも可能です。 

### EC2 / ECS on EC2 を選ぶ条件

* **GPU付きなど特殊インスタンスが必要**
* **特殊OSでコンテナ化が困難**
* つまり **サーバーレスで実現困難な場合の最後の選択肢**です。

## 実務での判断をさらに噛み砕くと

### Lambda 向き

* 1リクエストごとの単機能処理
* ファイルアップロード後の検証
* DB更新後の通知
* 差分処理
* 小さめのAPIバックエンド  

### Fargate 向き

* 長時間ジョブ
* 既存ツールを含むコンテナ化可能な処理
* 常駐型Webバックエンド
* イベントを受けて起動するコンテナジョブ 

### EC2 向き

* Windows依存や特殊OS依存が強い
* 既存ミドルがコンテナに落ちない
* 特殊デバイスやライセンス制約がある 

## 参画前に最低限持つべき感覚

* **まずLambdaで切れないか考える**
* 無理なら **Fargateに寄せられないか考える**
* どうしても無理なときだけ **EC2理由を説明できる**ようにする
  この順番です。

# 3. バッチ前提からイベントドリブン寄りへ

## なぜそうするのか

GCASは、従来の夜間バッチには次の問題があるとしています。

* 夜間停止（閉局）が必要になりやすい
* 夜間監視要員が必要
* バッチ失敗時に朝までの復旧が難しい
* ジョブ管理サーバや複雑なジョブ連携が必要
* 結果として運用が重くなる 

その代わりに、**更新イベントが起きるたびに後続処理を進める**イベントドリブンに寄せると、

* 運用がシンプルになる
* リトライ / 再処理がシンプル
* 常に最新データを出しやすい
* UXが上がる
  と整理しています。 

## どういう構成になるのか

GCASの運用マネージド化資料では、イベントドリブンの基本要素を

* **イベントプロデューサ**
* **イベントルータ / イベントバス / イベントストア**
* **イベントコンシューマ**

に分けています。対応サービス例として、

* イベントルータ側：**SQS / EventBridge / Amazon MQ**
* コンシューマ側：**Lambda / ECS Task / EKS Pod / AWS Batch**
  が挙がっています。

さらに実装例として、

* **SNS → SQS → Lambda**
* **EventBridge → Lambda 等**
* **DB差分ストリーム → Lambda → Firehose → 保存先**
* **S3アップロードイベント → Lambda → Glue**

のような構成が示されています。  

## ここで勘違いしやすい点

**「バッチを完全廃止」ではありません。**
GCAS自身も、ある時点のデータで一括処理が必要なケースでは **バッチを完全になくすことはできない** と明記しています。
ただし、**“夜間まとめて一気にやるしかない” 部分を減らす**のが方向性です。

## 参画前に理解しておくべきこと

* イベントドリブンは、**トリガーごとに細かく処理する設計**
* バッチより **疎結合** にしやすい
* **再実行・部分失敗対応** を簡単にしやすい
* ただし **整合性の扱い** は別途考える必要がある（CQRS など）

# 4. 現行オンプレ構成を見て「そのまま / マネージドへ / 要改修」を分ける考え方

## まず事実として

デジタル庁の「クラウド最適なアーキテクチャのサンプル」では、従来の3層Webアプリは

* Webアプリサーバ
* RDB
* 夜間バッチ
* ジョブ管理サーバ
* 監視サーバ
* ログ収集サーバ
* セキュリティ管理サーバ
* 踏み台
* パッチ管理
* アンチウイルス管理
  などを抱えがちだと整理しています。 

一方、クラウド最適化に寄せると、**OSを直接管理する対象が減り、踏み台・パッチ管理・ウイルス対策管理のような周辺サーバも不要になりやすい**としています。

## ここからの実務的な見分け方

ここは一部 **推論** です。
ただ、現場でかなり使える雑な切り分けは次です。

### そのまま寄り

* 業務ロジック自体は大きく変えなくてよい
* まずはAPI化や外部化を最小限で入れれば動く
* 既存DBスキーマを大きく崩さなくてよい
  → 典型例：単純な業務API、既存RDBの一部継続利用

### マネージドへ寄せる対象

* DB
* ログ
* 監視
* バックアップ
* ジョブスケジューラ
* メッセージング
* 共有ストレージ
  → ここはGCASが強く置換を推している部分です。  

### 要改修になりやすい対象

* セッションをサーバメモリに持つアプリ
* 夜間バッチ前提
* サーバローカルディスク前提
* スティッキーセッション前提
* 特殊OS / 特殊ミドル強依存
* サーバに手で入って運用する前提
  → これらはステートレス化、イベントドリブン化、S3化、コンテナ化を阻害しやすいです。 

## 現場での見方

オンプレ構成図を見たら、まずこの3つで色分けすると実務で強いです。

* **状態を持っている場所**
* **夜間にしか動かない処理**
* **人手で運用している場所**

この3つが濃いほど、「要改修」になりやすいです。
これは GCAS のモダン化定義そのものと一致します。

# 5. セッション外出し / ファイル外出し / ジョブ分離 を移行論点として拾う

## 5-1. セッション外出し

GCASのクラウドサービス活用資料では、**セッション情報はサーバ外部に保持する必要がある** とかなり明確に書かれています。理由は、

* オートスケールで台数が変わっても継続利用できる
* どのサーバでも応答できる
* スティッキーセッション不要になる
  からです。保存先候補として **ElastiCache / DynamoDB / RDS / MemoryDB** が挙げられています。さらに、**複数処理にまたがるトランザクション状態をセッションに持つべきではない** としています。 

### つまり実務では

* “ログイン状態をどこに持つか”
* “途中入力状態をどこに置くか”
* “ロードバランサ配下でどのノードに当たっても動くか”
  を見る必要があります。
  ここが曖昧だと、スケールも可用性も崩れやすいです。

---

## 5-2. ファイル外出し

共有ストレージのオブジェクトストレージ化資料では、**中間ファイルや保管ファイルを S3 に寄せる**ことを前提にしています。
そのとき重要なのは、**S3に置いたら終わりではなく、オブジェクトキーをDB等に保管する必要がある**ことです。
また、中間ファイルのような短期データは **S3ライフサイクルで自動削除**を検討するよう推奨しています。

デジタル庁のコスト削減記事でも、**ファイルをオブジェクトストレージに置けるなら大きくコストを下げやすい** としています。

### つまり実務では

* サーバローカル保存前提か
* 共有フォルダ前提か
* オブジェクトキー管理があるか
* 削除ポリシーがあるか
  を確認します。
  ファイルをサーバに持ったままだと、スケールや障害復旧が重くなりやすいです。

---

## 5-3. ジョブ分離

GCASでは、ジョブ管理を

* **スケジューラ / オーケストレータ**
* **実行コンピュート**

に分けて考えています。
スケジューラは **EventBridge Scheduler 推奨**、ETL は **Glue**、一般的なジョブフローは **Step Functions** が有力です。実行側は **Lambda / Fargate / EC2** の順に考えます。 ([AWS ドキュメント][3])

### つまり実務では

“1本の巨大バッチ” を見つけたら、

* **スケジュール管理**
* **実行本体**
* **後続通知**
* **再実行単位**
* **保存先**

に分けて見ます。
これができると、現行の重いジョブを **イベント分割** しやすくなります。

# あなた向けに一番大事な見方だけ絞ると

## 必須で持つべき判断軸

* **重い処理はAPI本体から切り離せるか**
* **状態をサーバ内に持っていないか**
* **ファイルをサーバローカルや共有フォルダに置いていないか**
* **夜間一括処理をイベントに分解できないか**
* **Lambdaで足りるか → Fargateに寄せられるか → それでも無理ならEC2か**

## 推奨で持つと強い判断軸

* **その機能は“更新系”か“参照系”か** を分けて見られる（CQRSの入口）
* **ストリーム / イベント / オブジェクトストレージ** を使った後続処理の形を想像できる
* **運用対象サーバを減らせる箇所** を構成図から拾える

# 最後にひとこと

今回の5項目を一言でまとめると、
**「サーバに状態と運用を抱え込んだオンプレ前提の作りを、イベントとマネージドで疎結合にほどいていく」**
という話です。

Replatform であっても、GCAS は **“そのまま移す” ではなく “最低限、運用が軽くなる方向に寄せる”** ことをかなり強く求めています。 ([GCASガイド][1])
