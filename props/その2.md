## 第2回 📘

今回は、前回の続きとして **RDS / Secrets Manager / AWS Backup / SSM** を、前回と同じ形で整理します。

---

## 結論 📌

**Windows オンプレ → AWS の Replatform 実務**で、この回で特に外しにくいのは次です。

* **RDS**

  * `rds.DatabaseInstance`
  * `rds.SubnetGroup`
* **Secrets Manager**

  * `secretsmanager.Secret`
  * `secretsmanager.RotationSchedule`
* **AWS Backup**

  * `backup.BackupVault`
  * `backup.BackupPlan`
  * `backup.BackupSelection`
* **SSM**

  * `ssm.StringParameter`
  * `ssm.CfnAssociation`
  * `ssm.CfnMaintenanceWindow`
  * `ssm.CfnMaintenanceWindowTask`
  * `ssm.CfnPatchBaseline`

**事実**として、GCAS では **DB のマネージドサービス化は必須で検討し、阻害要因がなければ適用推奨**、また **監視も必須で検討し、阻害要因がなければ適用推奨** です。さらに、設定パラメータは **サーバ外部に保存**し、保管先として **Systems Manager Parameter Store** と **AWS Secrets Manager** が例示されています。  

**推論**として、この前提だと
**「EC2 ローカル設定や DB ネイティブ機能に寄せる」より、「RDS / SSM / Secrets Manager / AWS Backup に寄せる」props が優先**です。

# 1. RDS 🗄️

GCAS では、データベースのマネージドサービス化は必須検討で、バックアップ・監視・メンテナンスもマネージド機能への置き換えが例示されています。可用性向上では、RDS の **Multi-AZ** が明示されています。

## 1-1. `rds.DatabaseInstance`

以下の prop 名・型・必須/任意は、AWS 公式 `DatabaseInstanceProps` 準拠です。`vpc` は必須で、`engine`、`allocatedStorage`、`credentials`、`storageEncrypted`、`monitoringInterval`、`multiAz`、`cloudwatchLogsExports`、`preferredMaintenanceWindow` などが定義されています。([AWS ドキュメント][1])

### `vpc`

* **型**: `ec2.IVpc`
* **必須か**: **必須**
* **何に使う props か**: DB をどの VPC に置くか決める
* **よく使う値（推論）**: 事前作成した `vpc`
* **その値にする理由（推論）**: subnet / SG / endpoint / 接続経路の土台になるため

### `engine`

* **型**: `IInstanceEngine`
* **必須か**: **必須**
* **何に使う props か**: DB エンジン種別を決める
* **よく使う値（推論）**:

  * 現行踏襲寄りなら `rds.DatabaseInstanceEngine.sqlServerSe(...)`
  * 脱 Microsoft も視野なら `rds.DatabaseInstanceEngine.postgres(...)`
* **その値にする理由（推論）**: GCAS でも「現行踏襲を大前提にせず、運用要件・コスト・メンテ性で選ぶ」考え方だからです。

### `instanceType`

* **型**: `ec2.InstanceType`
* **必須か**: 任意
* **何に使う props か**: DB インスタンスのサイズを決める
* **よく使う値（推論）**: `db.m6i.large` 相当から開始
* **その値にする理由（推論）**: Windows 系業務の DB はメモリ不足が起きやすく、小さすぎる初期値は事故になりやすいため

### `vpcSubnets`

* **型**: `ec2.SubnetSelection`
* **必須か**: 任意
* **何に使う props か**: どの subnet 群に DB を置くか決める
* **よく使う値（推論）**: `{ subnetGroupName: 'private-db' }` または isolated/private DB subnet
* **その値にする理由（推論）**: DB は外部公開せず、アプリ層からのみ到達させるのが基本だからです

### `securityGroups`

* **型**: `ec2.ISecurityGroup[]`
* **必須か**: 任意
* **何に使う props か**: DB に適用する SG を決める
* **よく使う値（推論）**: `[dbSg]`
* **その値にする理由（推論）**: ALB や踏み台ではなく、アプリ SG からだけ許可する構成にしやすいため

### `credentials`

* **型**: `Credentials`
* **必須か**: 任意
* **何に使う props か**: DB の管理者認証情報を決める
* **よく使う値（推論）**: `rds.Credentials.fromGeneratedSecret('admin')`
* **その値にする理由（推論）**: 認証情報をコード直書きせず、Secrets Manager と自然につなげやすいため

### `databaseName`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: 初期 DB 名を指定する
* **よく使う値（推論）**: 業務用途が分かる短い名前
* **その値にする理由（推論）**: 初期構築時の自動化と運用説明がしやすくなるため

### `allocatedStorage`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: 初期ストレージ容量を決める
* **よく使う値（推論）**: `100` 以上から開始
* **その値にする理由（推論）**: 移行直後はバックアップ・一時領域・監査ログで想定より増えやすいため

### `maxAllocatedStorage`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: ストレージ自動拡張の上限を決める
* **よく使う値（推論）**: `200`〜`500`
* **その値にする理由（推論）**: 容量逼迫の一次回避をしつつ、無制限拡張も避けたいからです

### `storageType`

* **型**: `StorageType`
* **必須か**: 任意
* **何に使う props か**: DB ストレージ種別を決める
* **よく使う値（推論）**: `rds.StorageType.GP3`
* **その値にする理由（推論）**: 現在の一般的な初期選択として、性能/コストのバランスを取りやすいため

### `storageEncrypted`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: ストレージ暗号化の有無を決める
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 行政系・業務系の DB では暗号化を前提にしておくほうが説明しやすいため

### `storageEncryptionKey`

* **型**: `kms.IKey`
* **必須か**: 任意
* **何に使う props か**: DB ストレージ暗号化に使う KMS キーを決める
* **よく使う値（推論）**: 事前作成した `kmsKey`
* **その値にする理由（推論）**: どの鍵で暗号化しているかを明確にできるため

### `multiAz`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: Multi-AZ 構成にするか決める
* **よく使う値（推論）**: 本番は `true`
* **その値にする理由（推論）**: GCAS でも可用性向上手段として Multi-AZ を明示しているためです。

### `backupRetention`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: 自動バックアップ保持期間を決める
* **よく使う値（推論）**: `cdk.Duration.days(7)` 〜 `35`
* **その値にする理由（推論）**: 短すぎると復旧余地がなく、長すぎるとコストが増えるため、要件に合わせて現実的な範囲で置くため

### `preferredBackupWindow`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: バックアップを取りたい時間帯を決める
* **よく使う値（推論）**: 業務影響の少ない深夜帯
* **その値にする理由（推論）**: 日中ピークの I/O 競合を避けやすいため

### `preferredMaintenanceWindow`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: メンテナンス時間帯を決める
* **よく使う値（推論）**: 深夜〜早朝の固定枠
* **その値にする理由（推論）**: RDS はメンテナンスウィンドウによる自動適用が前提になりやすいからです。

### `monitoringInterval`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: Enhanced Monitoring 間隔を決める
* **よく使う値（推論）**: `cdk.Duration.seconds(60)`
* **その値にする理由（推論）**: 詳細すぎる間隔はコスト増になりやすく、まず 60 秒が扱いやすいため

### `enablePerformanceInsights`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: Performance Insights を有効化する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: SQL ボトルネック調査で効きやすく、移行直後の性能分析に役立つため

### `performanceInsightRetention`

* **型**: `PerformanceInsightRetention`
* **必須か**: 任意
* **何に使う props か**: PI の保持期間を決める
* **よく使う値（推論）**: まずは短〜中期保持
* **その値にする理由（推論）**: まずは移行後の安定化期間を見られればよく、過剰保持を避けやすいため

### `cloudwatchLogsExports`

* **型**: `string[]`
* **必須か**: 任意
* **何に使う props か**: CloudWatch Logs に出す DB ログ種別を決める
* **よく使う値（推論）**: エラーログ / 監査系ログ中心
* **その値にする理由（推論）**: GCAS でも監視・ログ活用を重視しているため、まず障害切り分けに必要なログから出すのが実務的です。

### `cloudwatchLogsRetention`

* **型**: `logs.RetentionDays`
* **必須か**: 任意
* **何に使う props か**: CloudWatch Logs の保持期間を決める
* **よく使う値（推論）**: `THREE_MONTHS` など
* **その値にする理由（推論）**: 保持未設定だとコストも説明も曖昧になりやすいため

### `deletionProtection`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: DB の誤削除を防ぐ
* **よく使う値（推論）**: 本番は `true`
* **その値にする理由（推論）**: DB 削除事故の影響が最も大きい層のひとつだからです

### `publiclyAccessible`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: public に出すかどうかを決める
* **よく使う値（推論）**: `false`
* **その値にする理由（推論）**: 業務 DB は private 配置が基本だからです

### `parameterGroup`

* **型**: `IParameterGroup`
* **必須か**: 任意
* **何に使う props か**: DB パラメータ調整を外出しする
* **よく使う値（推論）**: 明示作成して渡す
* **その値にする理由（推論）**: 既定値依存より、差分をコードレビューしやすいため

### `removalPolicy`

* **型**: `cdk.RemovalPolicy`
* **必須か**: 任意
* **何に使う props か**: スタック削除時の扱いを決める
* **よく使う値（推論）**: `cdk.RemovalPolicy.SNAPSHOT` または `RETAIN`
* **その値にする理由（推論）**: 本番 DB を消し切るより、まず復旧余地を残すほうが安全だからです

### `subnetGroupName`

# 2. Secrets Manager 🔐

GCAS では、サーバ設定パラメータは **サーバ外部に保存**し、保管先として **Parameter Store** と **Secrets Manager** を例示しています。Secrets Manager は **シークレット専用ストアで、ライフサイクル管理や自動ローテーション**が特徴です。

## 2-1. `secretsmanager.Secret`

以下の prop 名・型・必須/任意は、AWS 公式 `SecretProps` 準拠です。`description`、`encryptionKey`、`generateSecretString`、`replicaRegions`、`removalPolicy`、`secretName` などが定義されています。なお、`secretStringValue` を synth 時に入れるのは、テンプレート露出の観点から AWS 公式でも強く非推奨です。([AWS ドキュメント][3])

### `secretName`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: シークレット名を決める
* **よく使う値（推論）**: `/app/prod/db/admin`
* **その値にする理由（推論）**: 環境・用途・対象を階層で見分けやすくするため

### `description`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: シークレットの用途説明
* **よく使う値（推論）**: `'Admin credential for application database'`
* **その値にする理由（推論）**: 同種の secret が増えたときに追いやすいからです

### `encryptionKey`

* **型**: `kms.IKey`
* **必須か**: 任意
* **何に使う props か**: シークレット暗号化に使う KMS キーを決める
* **よく使う値（推論）**: 事前作成した `kmsKey`
* **その値にする理由（推論）**: どの CMK で守るかを明確にできるため

### `generateSecretString`

* **型**: `SecretStringGenerator`
* **必須か**: 任意
* **何に使う props か**: ランダムな secret 値の生成方法を決める
* **よく使う値（推論）**: パスワード自動生成 + 必要な除外文字設定
* **その値にする理由（推論）**: 手入力パスワードより安全で、初期構築の自動化にも向くため

### `secretStringValue`

* **型**: `SecretValue`
* **必須か**: 任意
* **何に使う props か**: 初期 secret 値を直接指定する
* **よく使う値（推論）**: **原則使わない**
* **その値にする理由（推論）**: synth 出力やテンプレート露出のリスクを避けたいからです。([AWS ドキュメント][4])

### `secretObjectValue`

* **型**: `{ [key: string]: SecretValue }`
* **必須か**: 任意
* **何に使う props か**: JSON 形式で secret を持たせる
* **よく使う値（推論）**: DB 接続情報を JSON で保持
* **その値にする理由（推論）**: username / password / endpoint をひとまとまりで扱いやすいため

### `replicaRegions`

* **型**: `ReplicaRegion[]`
* **必須か**: 任意
* **何に使う props か**: 別リージョンへ secret を複製する
* **よく使う値（推論）**: 単一リージョン運用なら未設定
* **その値にする理由（推論）**: まずは東京単独で始める案件が多く、DR 要件が固まってからで十分なため

### `removalPolicy`

* **型**: `cdk.RemovalPolicy`
* **必須か**: 任意
* **何に使う props か**: スタック削除時の扱いを決める
* **よく使う値（推論）**: `cdk.RemovalPolicy.RETAIN`
* **その値にする理由（推論）**: 誤削除で運用不能になる影響が大きいため

## 2-2. `secretsmanager.RotationSchedule`

AWS 公式では、`RotationScheduleProps` に `secret`、`automaticallyAfter`、`hostedRotation`、`rotationLambda`、`rotateImmediatelyOnUpdate` などがあります。`automaticallyAfter` の既定は **30 日**です。([AWS ドキュメント][5])

### `secret`

* **型**: `ISecret`
* **必須か**: **必須**
* **何に使う props か**: どの secret をローテーションするか決める
* **よく使う値（推論）**: 事前作成した `dbSecret`
* **その値にする理由（推論）**: DB 認証情報とローテーション設定を結び付けるため

### `automaticallyAfter`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: ローテーション間隔を決める
* **よく使う値（推論）**: `cdk.Duration.days(30)`
* **その値にする理由（推論）**: AWS 公式の既定に近く、運用負荷とセキュリティのバランスを取りやすいため

### `hostedRotation`

* **型**: `HostedRotation`
* **必須か**: 任意
* **何に使う props か**: AWS 管理の回転ロジックを使う
* **よく使う値（推論）**: 対応 DB があるなら hosted rotation
* **その値にする理由（推論）**: 自前 Lambda より構築/保守負荷を抑えやすいため

### `rotationLambda`

* **型**: `lambda.IFunction`
* **必須か**: 任意
* **何に使う props か**: 独自ローテーション Lambda を使う
* **よく使う値（推論）**: hosted で足りないときだけ
* **その値にする理由（推論）**: 独自実装は柔軟だが、保守責任も増えるため

### `rotateImmediatelyOnUpdate`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: 設定変更時に即時ローテーションするか決める
* **よく使う値（推論）**: `false`
* **その値にする理由（推論）**: 変更のたびに即時ローテーションすると切替影響が読みづらくなるため

# 3. AWS Backup 💾

GCAS では、DB 製品固有バックアップの代替として **AWS Backup** が明示されています。また、災害対策レベルの例でも **AWS Backup / RDS Snapshot / S3 Replication** が出ています。

## 3-1. `backup.BackupVault`

`BackupVaultProps` には `backupVaultName`、`encryptionKey`、`blockRecoveryPointDeletion`、`lockConfiguration`、`notificationTopic`、`notificationEvents`、`removalPolicy`、`accessPolicy` などがあります。`removalPolicy` の既定は **RETAIN** です。([AWS ドキュメント][6])

### `backupVaultName`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: backup vault 名を決める
* **よく使う値（推論）**: `app-prod-backup-vault`
* **その値にする理由（推論）**: 復旧時にどの vault か識別しやすくするため

### `encryptionKey`

* **型**: `kms.IKey`
* **必須か**: 任意
* **何に使う props か**: バックアップ保護用の KMS キーを決める
* **よく使う値（推論）**: 専用の `backupKmsKey`
* **その値にする理由（推論）**: 本体データ鍵と分けると管理責任を整理しやすいため

### `blockRecoveryPointDeletion`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: recovery point の削除を抑止する
* **よく使う値（推論）**: 本番は `true` を検討
* **その値にする理由（推論）**: 誤削除耐性を高めやすいため

### `lockConfiguration`

* **型**: `LockConfiguration`
* **必須か**: 任意
* **何に使う props か**: Vault Lock を有効化する
* **よく使う値（推論）**: 本番の厳格運用で有効化検討
* **その値にする理由（推論）**: 保持期間の改ざん・削除を抑えたい場合に効くため

### `notificationTopic`

* **型**: `sns.ITopic`
* **必須か**: 任意
* **何に使う props か**: バックアップイベント通知先を決める
* **よく使う値（推論）**: 運用通知用 SNS Topic
* **その値にする理由（推論）**: バックアップ失敗を埋もれさせないため

### `notificationEvents`

* **型**: `BackupVaultEvents[]`
* **必須か**: 任意
* **何に使う props か**: 通知対象イベントを選ぶ
* **よく使う値（推論）**: 失敗・期限切れ・コピー失敗中心
* **その値にする理由（推論）**: 全通知より、まず障害系を優先したほうが運用しやすいため

### `removalPolicy`

* **型**: `cdk.RemovalPolicy`
* **必須か**: 任意
* **何に使う props か**: スタック削除時の扱いを決める
* **よく使う値（推論）**: `cdk.RemovalPolicy.RETAIN`
* **その値にする理由（推論）**: vault ごと消してしまう事故を避けたいからです

### `accessPolicy`

* **型**: `iam.PolicyDocument`
* **必須か**: 任意

# 4. SSM 🛠️

GCAS では、サーバ設定パラメータはサーバ外部に保存し、Parameter Store / Secrets Manager を使うことが示されています。Windows 運用を踏み台レス・手作業レスに寄せるなら、SSM はかなり重要です。

AWS 公式 `StringParameterProps` には `stringValue` が必須で、`parameterName`、`description`、`allowedPattern`、`dataType`、`type`、`tier` などがあります。([AWS ドキュメント][9])

* **何に使う props か**: String/他形式を決める
* **よく使う値（推論）**: 非機密なら既定の `STRING`
* **その値にする理由（推論）**: 機密値をここに混ぜすぎず、責務分離しやすいため
