# 1. ネットワーク 🌐

## 1-1. `ec2.Vpc`

以下の prop 名・型・必須/任意は AWS 公式の `Vpc` / `VpcProps` ベースです。`Vpc` は VPC CIDR を分割し、AZ ごとにサブネットや NAT/IGW 周りを自動構成できます。 ([AWS ドキュメント][2])

### `ipAddresses`

* **型**: `IIpAddresses`
* **必須か**: 任意
* **何に使うか**: VPC の IPv4 アドレス空間を決める
* **よく使う値（推論）**: `ec2.IpAddresses.cidr('10.20.0.0/16')`
* **その値にする理由（推論）**: `/16` だと複数 AZ・複数層・将来増設まで見込みやすく、Windows リプレイスで後から subnet を切り増ししやすい

### `maxAzs`

* **型**: `number`
* **必須か**: 任意
* **何に使うか**: 使う AZ 数の上限を決める
* **よく使う値（推論）**: `2`
* **その値にする理由（推論）**: 可用性を確保しつつ、最初から 3 AZ に広げて複雑化・高コスト化しすぎないため

### `natGateways`

* **型**: `number`
* **必須か**: 任意
* **何に使うか**: private subnet から外向き通信する NAT の台数を決める
* **よく使う値（推論）**: `1` または `2`
* **その値にする理由（推論）**: 検証/小規模は `1` でコスト抑制、本番で AZ 障害時の経路維持を強めたいなら `2` に寄せやすい

### `subnetConfiguration`

* **型**: `SubnetConfiguration[]`
* **必須か**: 任意
* **何に使うか**: subnet の種類と mask を AZ ごとにどう切るか決める
* **よく使う値（推論）**:

  * `PUBLIC`
  * `PRIVATE_WITH_EGRESS`
  * `PRIVATE_ISOLATED`
* **その値にする理由（推論）**: ALB・アプリ・DB を素直に 3 層分離でき、Windows リプレイスでも説明しやすい標準形になるため

### `flowLogs`

* **型**: `{ [string]: FlowLogOptions }`
* **必須か**: 任意
* **何に使うか**: VPC Flow Logs を有効化する
* **よく使う値（推論）**: CloudWatch Logs 出力を 1 つ定義
* **その値にする理由（推論）**: 通信トラブルや監査の切り分けで「まず見るログ」になりやすいから

### `gatewayEndpoints`

* **型**: `{ [string]: GatewayVpcEndpointOptions }`
* **必須か**: 任意
* **何に使うか**: S3 / DynamoDB などへ VPC Endpoint 経由で到達させる
* **よく使う値（推論）**: `S3` endpoint を追加
* **その値にする理由（推論）**: ログ退避・バックアップ・資材取得で S3 を使う場面が多く、NAT 依存と通信コストを減らしやすい

### `enableDnsSupport`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: VPC 内の DNS 解決を有効化する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: SSM・各種 AWS エンドポイント・内部名前解決で困りやすいため、無効化する理由が薄い

### `enableDnsHostnames`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: インスタンスの DNS ホスト名付与を有効化する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 運用確認や一部サービス連携で hostname があったほうが扱いやすい

### `restrictDefaultSecurityGroup`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: default SG の緩い初期 inbound/outbound を削除する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 「default SG 放置」を避け、最小権限の説明をしやすくするため

### `natGatewaySubnets`

* **型**: `SubnetSelection`
* **必須か**: 任意
* **何に使うか**: NAT をどの public subnet 群に置くか決める
* **よく使う値（推論）**: `{ subnetGroupName: 'public-ingress' }`
* **その値にする理由（推論）**: public subnet が複数グループある場合でも、NAT の置き場所を明示できて事故を減らせる

## 1-2. `ec2.SecurityGroup`

以下は AWS 公式の `SecurityGroup` / `SecurityGroupProps` ベースです。`vpc` は必須で、`allowAllOutbound` の既定値は `true` です。 ([AWS ドキュメント][3])

### `vpc`

* **型**: `IVpc`
* **必須か**: **必須**
* **何に使うか**: どの VPC に属する Security Group か指定する
* **よく使う値（推論）**: 事前作成した `vpc`
* **その値にする理由（推論）**: SG は VPC に属する前提なので、まずここが起点になるため

### `allowAllOutbound`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: IPv4 outbound を全許可するか決める
* **よく使う値（推論）**: `false`
* **その値にする理由（推論）**: Windows サーバでも、通信先を絞る設計のほうが後で監査・説明しやすい

### `allowAllIpv6Outbound`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: IPv6 outbound を全許可するか決める
* **よく使う値（推論）**: 未設定、または `false`
* **その値にする理由（推論）**: 最初から IPv6 を使わない構成なら、余計な許可面を増やさないほうが安全

### `description`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: SG の役割を説明する
* **よく使う値（推論）**: `'Security group for public ALB'`
* **その値にする理由（推論）**: コンソールや監査時に用途が一目で分かるから

### `disableInlineRules`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: ルールを SG リソースに inline で持たせるか、別リソース化しやすくするか決める
* **よく使う値（推論）**: `true` を検討
* **その値にする理由（推論）**: 差分レビューやルール単位の追跡を重視する現場では分離のほうが見やすい

### `securityGroupName`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: 物理名を固定する
* **よく使う値（推論）**: 原則未設定
* **その値にする理由（推論）**: 物理名固定は更新時の自由度を落としやすく、固定理由がある時だけ使うほうが無難

## 1-3. `elbv2.ApplicationLoadBalancer`

以下は AWS 公式の `ApplicationLoadBalancer` / `ApplicationLoadBalancerProps` ベースです。`vpc` は必須で、`securityGroup` は未指定だと自動作成されます。 ([AWS ドキュメント][4])

### `vpc`

* **型**: `IVpc`
* **必須か**: **必須**
* **何に使うか**: ALB の配置先 VPC を決める
* **よく使う値（推論）**: 事前作成した `vpc`
* **その値にする理由（推論）**: ALB は VPC 配置が前提で、subnet や SG 設計の土台になるため

### `internetFacing`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: インターネット公開 ALB か internal ALB か決める
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 住民向け/外部向け Web 入口として使うなら public ALB が基本になるため

### `securityGroup`

* **型**: `ISecurityGroup`
* **必須か**: 任意
* **何に使うか**: ALB 自体に適用する SG を明示する
* **よく使う値（推論）**: 明示作成した `albSg`
* **その値にする理由（推論）**: 自動作成任せより、受け口ルールをレビューしやすい

### `vpcSubnets`

* **型**: `SubnetSelection`
* **必須か**: 任意
* **何に使うか**: どの subnet に ALB を載せるか決める
* **よく使う値（推論）**: `{ subnetType: ec2.SubnetType.PUBLIC }`
* **その値にする理由（推論）**: public ALB は public subnet に明示配置したほうが意図がぶれない

### `idleTimeout`

* **型**: `Duration`
* **必須か**: 任意
* **何に使うか**: アイドル接続のタイムアウトを決める
* **よく使う値（推論）**: `cdk.Duration.seconds(60)`
* **その値にする理由（推論）**: 既定値のままでもよいが、業務 API の待ち時間要件に合わせて意図的に固定したほうが運用しやすい

### `deletionProtection`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: ALB の誤削除を防ぐ
* **よく使う値（推論）**: 本番は `true`
* **その値にする理由（推論）**: 入口の LB を消す事故は影響が大きく、削除保護の効果が大きい

### `dropInvalidHeaderFields`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: 不正な HTTP ヘッダを破棄する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 入口で変なヘッダを落としたほうが、後段のアプリ実装に余計な負担を残しにくい

### `ipAddressType`

* **型**: `IpAddressType`
* **必須か**: 任意
* **何に使うか**: ALB の IP 種別を決める
* **よく使う値（推論）**: `elbv2.IpAddressType.IPV4`
* **その値にする理由（推論）**: まず IPv4 前提で固める案件が多く、IPv6 は必要性が明確になってから追加しやすい

# 2. コンピュート / IAM / 暗号化 🔐

## 2-1. `ec2.Instance`

以下は AWS 公式 `Instance` / `InstanceProps` ベースです。`instanceType`・`machineImage`・`vpc` は必須で、`role` や `securityGroup` は未指定なら自動作成できます。 ([AWS ドキュメント][5])

### `instanceType`

* **型**: `InstanceType`
* **必須か**: **必須**
* **何に使うか**: CPU / メモリ / 世代を決める
* **よく使う値（推論）**: `new ec2.InstanceType('m6i.large')` など
* **その値にする理由（推論）**: Windows はメモリ消費が軽くないため、最初から小さすぎるサイズを避けたほうが安全

### `machineImage`

* **型**: `IMachineImage`
* **必須か**: **必須**
* **何に使うか**: 起動する OS イメージを決める
* **よく使う値（推論）**: `ec2.MachineImage.latestWindows(...)`
* **その値にする理由（推論）**: Windows リプレイス前提では、OS 系列を明示しておかないと後で差異が大きくなるため

### `vpc`

* **型**: `IVpc`
* **必須か**: **必須**
* **何に使うか**: EC2 の配置先 VPC を決める
* **よく使う値（推論）**: 事前作成した `vpc`
* **その値にする理由（推論）**: NW・SG・subnet・endpoint 設計の土台になるため

### `vpcSubnets`

* **型**: `SubnetSelection`
* **必須か**: 任意
* **何に使うか**: どの subnet に置くか決める
* **よく使う値（推論）**: `{ subnetType: ec2.SubnetType.PRIVATE_WITH_EGRESS }`
* **その値にする理由（推論）**: Windows サーバは基本 private に置き、外部公開は ALB 側に寄せるほうが安全

### `securityGroup`

* **型**: `ISecurityGroup`
* **必須か**: 任意
* **何に使うか**: EC2 に適用する SG を指定する
* **よく使う値（推論）**: 明示作成した `appSg`
* **その値にする理由（推論）**: 自動 SG より、許可通信をレビューしやすい

### `role`

* **型**: `IRole`
* **必須か**: 任意
* **何に使うか**: EC2 から AWS API を呼ぶ権限を与える
* **よく使う値（推論）**: SSM / CloudWatch Agent / S3 / KMS を考慮した明示ロール
* **その値にする理由（推論）**: Windows 運用では SSM・ログ送信・暗号化連携がほぼ早期に必要になるため

### `ssmSessionPermissions`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: Session Manager 利用に必要な権限を追加する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: 踏み台や RDP 依存を減らす運用に寄せやすいから

### `requireImdsv2`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: IMDSv2 を強制する
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: メタデータ取得の保護を強められ、セキュリティの初期値として扱いやすい

### `blockDevices`

* **型**: `BlockDevice[]`
* **必須か**: 任意
* **何に使うか**: ルート/追加 EBS のサイズや暗号化を決める
* **よく使う値（推論）**: ルートボリュームを明示し、暗号化・容量を指定
* **その値にする理由（推論）**: Windows は C ドライブ不足やログ/更新領域不足が起きやすいので最初に固定したほうが安全

### `disableApiTermination`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: EC2 の API 経由削除を防ぐ
* **よく使う値（推論）**: 本番は `true`
* **その値にする理由（推論）**: 単体 EC2 を誤削除すると影響が大きいため

### `detailedMonitoring`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: 詳細モニタリングを有効化する
* **よく使う値（推論）**: 要件がある時だけ `true`
* **その値にする理由（推論）**: 常時有効にするとコストも増えるため、必要性が明確な監視要件に合わせるほうがよい

## 2-2. `iam.Role`

以下は AWS 公式 `Role` / `RoleProps` ベースです。`assumedBy` は必須です。 ([AWS ドキュメント][6])

### `assumedBy`

* **型**: `IPrincipal`
* **必須か**: **必須**
* **何に使うか**: 誰がこの Role を引き受けるか決める
* **よく使う値（推論）**: `new iam.ServicePrincipal('ec2.amazonaws.com')`
* **その値にする理由（推論）**: EC2 用ロールなら、まず信頼主体を EC2 サービスに正しく固定する必要があるため

### `managedPolicies`

* **型**: `IManagedPolicy[]`
* **必須か**: 任意
* **何に使うか**: 既存の managed policy をまとめて付与する
* **よく使う値（推論）**: `AmazonSSMManagedInstanceCore` など必要最小限
* **その値にする理由（推論）**: SSM 接続や運用基盤は managed policy を足掛かりにすると構築が安定しやすい

### `inlinePolicies`

* **型**: `{ [string]: PolicyDocument }`
* **必須か**: 任意
* **何に使うか**: この Role 専用のポリシーを内包する
* **よく使う値（推論）**: S3 特定バケット / KMS 特定キーだけの最小権限
* **その値にする理由（推論）**: managed policy だけで足りない細かい権限制御を絞り込みやすい

### `permissionsBoundary`

* **型**: `IManagedPolicy`
* **必須か**: 任意
* **何に使うか**: Role に持たせられる権限の上限を決める
* **よく使う値（推論）**: 組織標準の boundary policy
* **その値にする理由（推論）**: 現場ごとの差があっても、越えてはいけない上限を統一しやすい

### `description`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: Role の用途を説明する
* **よく使う値（推論）**: `'Role for Windows application EC2'`
* **その値にする理由（推論）**: IAM は数が増えるので、用途説明がないと後で見分けづらい

### `maxSessionDuration`

* **型**: `Duration`
* **必須か**: 任意
* **何に使うか**: 一時認証セッションの最大継続時間を決める
* **よく使う値（推論）**: `cdk.Duration.hours(1)` 〜 `4`
* **その値にする理由（推論）**: 長すぎると統制上弱く、短すぎると運用作業がしづらいため

### `path`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: IAM リソースの論理パスを整理する
* **よく使う値（推論）**: `'/system/'` など
* **その値にする理由（推論）**: 大規模環境で役割のまとまりを見やすくするため

### `roleName`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: Role の物理名を固定する
* **よく使う値（推論）**: 原則未設定
* **その値にする理由（推論）**: 物理名固定は置換更新や再利用時の制約になりやすい

# 3. ストレージ / ログ 🗂️

## 3-1. `s3.Bucket`

以下は AWS 公式 `Bucket` / `BucketProps` ベースです。 `blockPublicAccess`、`encryption`、`enforceSSL`、`lifecycleRules` などが主要 prop です。 ([AWS ドキュメント][8])

### `blockPublicAccess`

* **型**: `BlockPublicAccess`
* **必須か**: 任意
* **何に使うか**: public access を遮断する
* **よく使う値（推論）**: `s3.BlockPublicAccess.BLOCK_ALL`

## 3-2. `logs.LogGroup`

以下は AWS 公式 `LogGroup` / `LogGroupProps` ベースです。`retention`、`encryptionKey`、`removalPolicy`、`deletionProtectionEnabled` などが主要 prop です。 ([AWS ドキュメント][9])

### `retention`

* **型**: `RetentionDays`
* **必須か**: 任意
* **何に使うか**: ログ保持期間を決める
* **よく使う値（推論）**: `logs.RetentionDays.THREE_MONTHS` など
* **その値にする理由（推論）**: 保持期間を固定しないと、コストと運用ルールが曖昧になりやすい

### `encryptionKey`

* **型**: `IKey`
* **必須か**: 任意
* **何に使うか**: LogGroup を暗号化する KMS キーを指定する
* **よく使う値（推論）**: 事前作成した `kmsKey`
* **その値にする理由（推論）**: ログも機微情報を含みうるため、暗号化主体を明確にする

### `logGroupName`

* **型**: `string`
* **必須か**: 任意
* **何に使うか**: LogGroup の物理名を固定する
* **よく使う値（推論）**: `/app/prod/windows/system`
* **その値にする理由（推論）**: エージェント設定や運用手順と結び付けやすい

### `removalPolicy`

* **型**: `RemovalPolicy`
* **必須か**: 任意
* **何に使うか**: スタック削除時にログを残すか決める
* **よく使う値（推論）**: `cdk.RemovalPolicy.RETAIN`
* **その値にする理由（推論）**: 障害調査や監査で後から必要になる可能性が高いから

### `deletionProtectionEnabled`

* **型**: `boolean`
* **必須か**: 任意
* **何に使うか**: LogGroup の削除保護を有効化する
* **よく使う値（推論）**: 本番は `true` を検討
* **その値にする理由（推論）**: ログ消失は復元不能なことが多く、防止効果が高い

### `logGroupClass`

* **型**: `LogGroupClass`
* **必須か**: 任意
* **何に使うか**: `STANDARD` / `INFREQUENT_ACCESS` を選ぶ
* **よく使う値（推論）**: `logs.LogGroupClass.STANDARD`
* **その値にする理由（推論）**: まずは標準クラスで運用し、アクセス頻度が低いと分かってから見直すほうが安全

### `dataProtectionPolicy`

* **型**: `DataProtectionPolicy`
* **必須か**: 任意
* **何に使うか**: ログ内データ保護ルールを定義する
* **よく使う値（推論）**: 個人情報や機微情報のマスキング方針がある場合のみ設定
* **その値にする理由（推論）**: 必要性が明確な時には有効だが、最初から過剰に複雑化しないほうが導入しやすい
