GCAS / ガバメントクラウド文脈では、**監視設定の確認** と **可用性構成の確認** が重要で、サマリーでも **外形監視・RUM・ダッシュボード・アラーム連携** が例示されています。
なので、この回は **「公開経路」+「可用性」+「監視」** に直結する props を優先して押さえるのが実務的です。 ([GCASガイド][1])

# 1. Auto Scaling 🚀

## 1-1. `autoscaling.AutoScalingGroup`

`AutoScalingGroupProps` では、`vpc` が必須で、`vpcSubnets`、`minCapacity`、`maxCapacity`、`desiredCapacity`、`launchTemplate`、`healthChecks`、`signals`、`updatePolicy` などが主要です。さらに `desiredCapacity` は **空欄推奨**、`vpcSubnets` の既定は **All Private subnets** です。 ([AWS ドキュメント][2])

### `vpc`

* **型**: `ec2.IVpc`
* **必須か**: **必須**
* **何に使う props か**: ASG をどの VPC に置くか決める
* **よく使う値（推論）**: 事前作成した `vpc`
* **その値にする理由（推論）**: subnet / SG / ALB / Route 53 との接続の土台になるため。 ([AWS ドキュメント][2])

### `vpcSubnets`

* **型**: `ec2.SubnetSelection`
* **必須か**: 任意
* **何に使う props か**: ASG インスタンスをどの subnet に置くか決める
* **よく使う値（推論）**: `{ subnetGroupName: 'private-app' }`
* **その値にする理由（推論）**: 既定でも private ですが、**どの private 層かを明示**したほうがレビューしやすいからです。 ([AWS ドキュメント][2])

### `minCapacity`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: 最低台数を決める
* **よく使う値（推論）**: 本番は `2`
* **その値にする理由（推論）**: 2 AZ 配置なら最低 2 台にして、片系障害時の継続性を持たせやすいためです。 ([AWS ドキュメント][2])

### `maxCapacity`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: 最大台数を決める
* **よく使う値（推論）**: `4` 〜 `6`
* **その値にする理由（推論）**: Windows はスケールアウトを無制限にせず、上限を決めてコスト暴走を防ぎやすくするためです。 ([AWS ドキュメント][2])

### `desiredCapacity`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: 初期台数を決める
* **よく使う値（推論）**: **原則未設定**
* **その値にする理由（推論）**: 公式でも、数値を入れると **毎回デプロイ時にその数へ戻される**ため、空欄推奨だからです。 ([AWS ドキュメント][2])

### `launchTemplate`

* **型**: `ec2.ILaunchTemplate`
* **必須か**: 任意
* **何に使う props か**: 起動定義を Launch Template に寄せる
* **よく使う値（推論）**: 明示作成した `launchTemplate`
* **その値にする理由（推論）**: Windows の AMI・UserData・IMDSv2・EBS 条件を **1 か所で再利用**しやすいからです。 ([AWS ドキュメント][2])

### `healthChecks`

* **型**: `autoscaling.HealthChecks`
* **必須か**: 任意
* **何に使う props か**: EC2 / 追加ヘルスチェックを定義する
* **よく使う値（推論）**: `autoscaling.HealthChecks.withAdditionalChecks({ ... })`
* **その値にする理由（推論）**: ALB 配下なら EC2 だけでなく **LB 側の健全性も見たい**ためです。 ([AWS ドキュメント][2])

### `signals`

* **型**: `autoscaling.Signals`
* **必須か**: 任意
* **何に使う props か**: 起動完了シグナル待ちを入れる
* **よく使う値（推論）**: `autoscaling.Signals.waitForAll({ timeout: ... })`
* **その値にする理由（推論）**: Windows は起動後に SSM / CW Agent / アプリ初期化が終わるまで時間がかかるため、**起動直後を成功扱いしない**ほうが安全です。 ([AWS ドキュメント][2])

### `updatePolicy`

* **型**: `autoscaling.UpdatePolicy`
* **必須か**: 任意
* **何に使う props か**: ASG の差し替え時の更新方法を決める
* **よく使う値（推論）**: `autoscaling.UpdatePolicy.rollingUpdate()`
* **その値にする理由（推論）**: Windows の更新は一括置換より **段階更新**のほうが事故を抑えやすいためです。 ([AWS ドキュメント][2])

---

## 実務メモ 🧭

**推論**ですが、Windows リプレイスでは **必ず ASG にするとは限りません**。
業務パッケージ、ノード固定前提、ライセンス、共有状態が強い場合は **単体 EC2 + ALB** で始まることも普通にあります。
ただし、**Web / AP 層を冪等に寄せられるなら ASG の価値は大きい**です。

# 2. Launch Template 🧱

## 2-1. `ec2.LaunchTemplate`

`LaunchTemplateProps` は **全 props 任意**ですが、実務では `machineImage`、`instanceType`、`blockDevices`、`requireImdsv2`、`role`、`securityGroup`、`userData` が重要です。`httpTokens` は既定で `OPTIONAL`、`requireImdsv2` を `true` にすると `httpTokens` は `required` になります。 ([AWS ドキュメント][3])

### `machineImage`

* **型**: `ec2.IMachineImage`
* **必須か**: 任意
* **何に使う props か**: 起動 AMI を決める
* **よく使う値（推論）**: `ec2.MachineImage.latestWindows(...)`
* **その値にする理由（推論）**: Windows 2019/2022 など **OS 系列を明示**したほうが、移行後の差異を減らせるためです。 ([AWS ドキュメント][3])

### `instanceType`

* **型**: `ec2.InstanceType`
* **必須か**: 任意
* **何に使う props か**: デフォルトのインスタンスタイプを決める
* **よく使う値（推論）**: `m6i.large` 前後
* **その値にする理由（推論）**: Windows はメモリ・更新処理が重めなので、最初から細すぎるサイズを避けたいからです。 ([AWS ドキュメント][3])

### `blockDevices`

* **型**: `ec2.BlockDevice[]`
* **必須か**: 任意
* **何に使う props か**: ルート / 追加 EBS の構成を決める
* **よく使う値（推論）**: ルート EBS を明示し、容量・暗号化を固定
* **その値にする理由（推論）**: Windows は C ドライブ不足で詰まりやすく、AMI 既定任せを避けたいからです。 ([AWS ドキュメント][3])

### `requireImdsv2`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: IMDSv2 を必須にする
* **よく使う値（推論）**: `true`
* **その値にする理由（推論）**: メタデータ保護を強められるので、初期値としてかなり置きやすいためです。 ([AWS ドキュメント][3])

### `role`

* **型**: `iam.IRole`
* **必須か**: 任意
* **何に使う props か**: EC2 に付ける IAM ロールを決める
* **よく使う値（推論）**: SSM / CloudWatch Agent / S3 / KMS 最小権限ロール
* **その値にする理由（推論）**: Windows 運用では **SSM / ログ / 暗号化連携**がほぼ必須になるためです。 ([AWS ドキュメント][3])

### `securityGroup`

* **型**: `ec2.ISecurityGroup`
* **必須か**: 任意
* **何に使う props か**: 起動インスタンスに付ける SG を決める
* **よく使う値（推論）**: 明示作成した `appSg`
* **その値にする理由（推論）**: 自動 SG より、**どの通信を開けるか**を差分レビューしやすいためです。 ([AWS ドキュメント][3])

### `userData`

* **型**: `ec2.UserData`
* **必須か**: 任意
* **何に使う props か**: 起動時初期化を入れる
* **よく使う値（推論）**: SSM Agent / CW Agent / 初期設定スクリプト
* **その値にする理由（推論）**: 手作業セットアップを減らし、再現性を持たせやすいためです。 ([AWS ドキュメント][3])

# 3. Route 53 🌍

## 3-1. `route53.HostedZone`

`HostedZoneProps` の主要 props は `zoneName`、`vpcs`、`queryLogsLogGroupArn` です。
`vpcs` を指定すると **private hosted zone** になります。 ([AWS ドキュメント][4])

### `zoneName`

* **型**: `string`
* **必須か**: **必須**
* **何に使う props か**: ドメイン名を決める
* **よく使う値（推論）**: `example.go.jp` / `internal.example.local` 相当の組織ドメイン
* **その値にする理由（推論）**: 公開名・内部名の起点になるため、最初に揺らさないほうがよいからです。 ([AWS ドキュメント][4])

### `vpcs`

* **型**: `ec2.IVpc[]`
* **必須か**: 任意
* **何に使う props か**: private hosted zone を関連付ける VPC を決める
* **よく使う値（推論）**: `[vpc]`
* **その値にする理由（推論）**: 内部向け名前解決だけを VPC 内に閉じたい場合に使いやすいためです。 ([AWS ドキュメント][4])

### `queryLogsLogGroupArn`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: Route 53 query logs の送信先を決める
* **よく使う値（推論）**: 専用 `LogGroup` の ARN
* **その値にする理由（推論）**: 名前解決トラブル時の証跡を残しやすいためです。 ([AWS ドキュメント][4])

---

## 3-2. `route53.ARecord`

`ARecordProps` では `zone` と `target` が中心で、`recordName`、`ttl`、`healthCheck` などを持てます。`recordName` の既定は **zone root**、`ttl` の既定は **30 分**です。 ([AWS ドキュメント][5])

### `zone`

* **型**: `route53.IHostedZone`
* **必須か**: **必須**
* **何に使う props か**: どの hosted zone にレコードを作るか決める
* **よく使う値（推論）**: 事前作成 / import した `hostedZone`
* **その値にする理由（推論）**: 同名の DNS を別 zone に誤登録しないためです。 ([AWS ドキュメント][5])

### `target`

* **型**: `route53.RecordTarget`
* **必須か**: **必須**
* **何に使う props か**: レコードの向き先を決める
* **よく使う値（推論）**: `route53.RecordTarget.fromAlias(new targets.LoadBalancerTarget(alb))`
* **その値にする理由（推論）**: 公開入口を ALB に寄せる構成では、alias で結ぶのが自然だからです。 ([AWS ドキュメント][5])

### `recordName`

* **型**: `string`
* **必須か**: 任意
* **何に使う props か**: サブドメイン名を決める
* **よく使う値（推論）**: `www` / `api` / `app`
* **その値にする理由（推論）**: 役割ごとに名前を分けると、公開経路の責務が見えやすくなるためです。 ([AWS ドキュメント][5])

### `ttl`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: DNS キャッシュ時間を決める
* **よく使う値（推論）**: `cdk.Duration.minutes(5)` または `30`
* **その値にする理由（推論）**: 切替頻度が高い移行期は短め、本番安定後は長めが扱いやすいためです。 ([AWS ドキュメント][5])

### `healthCheck`

* **型**: `route53.IHealthCheck`
* **必須か**: 任意
* **何に使う props か**: Route 53 側のヘルスチェックと連動させる
* **よく使う値（推論）**: フェイルオーバー設計時のみ明示
* **その値にする理由（推論）**: 単純な単一 ALB 構成では不要なことも多く、**DR / active-passive** で効いてくるためです。 ([AWS ドキュメント][5])

---

## 実務メモ 🧭

**推論**ですが、Windows リプレイスでは **Private Hosted Zone = AD DNS の完全代替** とは限りません。
**業務アプリ用の内部 FQDN** は Route 53 private hosted zone、**AD 連携の名前解決** は別設計、という切り分けになることがあります。

# 4. ACM 🔒

## 4-1. `acm.Certificate`

`CertificateProps` では `domainName` が中心で、`validation`、`subjectAlternativeNames`、`keyAlgorithm`、`transparencyLoggingEnabled`、`allowExport` を持てます。`validation` の既定は **email**、`keyAlgorithm` の既定は **RSA_2048**、`allowExport` の既定は **false** です。 ([AWS ドキュメント][6])

### `domainName`

* **型**: `string`
* **必須か**: **必須**
* **何に使う props か**: 証明書の対象 FQDN を決める
* **よく使う値（推論）**: `app.example.go.jp`
* **その値にする理由（推論）**: HTTPS リスナーの正引き先に合わせて、まず主ドメインを固定したいからです。 ([AWS ドキュメント][6])

### `validation`

* **型**: `acm.CertificateValidation`
* **必須か**: 任意
* **何に使う props か**: 証明書の検証方法を決める
* **よく使う値（推論）**: `acm.CertificateValidation.fromDns(hostedZone)`
* **その値にする理由（推論）**: CDK 管理と相性がよく、email 検証より自動化しやすいためです。 ([AWS ドキュメント][6])

### `subjectAlternativeNames`

* **型**: `string[]`
* **必須か**: 任意
* **何に使う props か**: 追加 SAN を持たせる
* **よく使う値（推論）**: `['www.example.go.jp', 'api.example.go.jp']`
* **その値にする理由（推論）**: 同一サイト群を 1 枚で扱いたい時に便利だからです。 ([AWS ドキュメント][6])

### `keyAlgorithm`

* **型**: `acm.KeyAlgorithm`
* **必須か**: 任意
* **何に使う props か**: 証明書の鍵アルゴリズムを決める
* **よく使う値（推論）**: まずは既定の `RSA_2048`
* **その値にする理由（推論）**: 互換性を優先しやすく、最初の選択として無難だからです。 ([AWS ドキュメント][6])

### `transparencyLoggingEnabled`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: Certificate Transparency へログするか決める
* **よく使う値（推論）**: 既定の `true`
* **その値にする理由（推論）**: 公開証明書の一般的運用に寄せやすいためです。 ([AWS ドキュメント][6])

### `allowExport`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: 証明書のエクスポート可否を決める
* **よく使う値（推論）**: `false`
* **その値にする理由（推論）**: ALB 用証明書なら、わざわざ持ち出せる必要が薄いためです。 ([AWS ドキュメント][6])

# 5. ALB Listener 🎧

## 5-1. `elbv2.ApplicationListener`

`ApplicationListenerProps` では `loadBalancer` が必須で、`protocol`、`port`、`certificates`、`defaultAction`、`defaultTargetGroups`、`open`、`sslPolicy` が重要です。HTTPS/TLS の場合、**certificate は 1 枚必須**です。 ([AWS ドキュメント][7])

### `loadBalancer`

* **型**: `elbv2.IApplicationLoadBalancer`
* **必須か**: **必須**
* **何に使う props か**: どの ALB に listener を付けるか決める
* **よく使う値（推論）**: 作成済み `alb`
* **その値にする理由（推論）**: listener は ALB に従属するため、まず親を明示する必要があるためです。 ([AWS ドキュメント][7])

### `protocol`

* **型**: `elbv2.ApplicationProtocol`
* **必須か**: 任意
* **何に使う props か**: HTTP / HTTPS を決める
* **よく使う値（推論）**: `elbv2.ApplicationProtocol.HTTPS`
* **その値にする理由（推論）**: 外部公開入口なら TLS 終端を ALB に置くのが基本だからです。 ([AWS ドキュメント][7])

### `port`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: listener ポートを決める
* **よく使う値（推論）**: `443`
* **その値にする理由（推論）**: 公開 Web 系は 443 が標準で、利用者側にも説明しやすいためです。 ([AWS ドキュメント][7])

### `certificates`

* **型**: `elbv2.IListenerCertificate[]`
* **必須か**: 任意
* **何に使う props か**: HTTPS/TLS 用の ACM 証明書を渡す
* **よく使う値（推論）**: `[elbv2.ListenerCertificate.fromArn(cert.certificateArn)]`
* **その値にする理由（推論）**: HTTPS listener を成立させるのに必要だからです。 ([AWS ドキュメント][7])

### `defaultAction`

* **型**: `elbv2.ListenerAction`
* **必須か**: 任意
* **何に使う props か**: デフォルト動作を決める
* **よく使う値（推論）**: `ListenerAction.forward([tg])`
* **その値にする理由（推論）**: 単純な 1 系統構成なら target group へ forward が最も分かりやすいためです。 ([AWS ドキュメント][7])

### `defaultTargetGroups`

* **型**: `elbv2.IApplicationTargetGroup[]`
* **必須か**: 任意
* **何に使う props か**: デフォルト転送先 TG を指定する
* **よく使う値（推論）**: 単純構成では 1 TG
* **その値にする理由（推論）**: ただし複雑な制御なら `defaultAction` 側のほうが柔軟だからです。 ([AWS ドキュメント][7])

### `open`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: listener port を広く開けるか決める
* **よく使う値（推論）**: `false`
* **その値にする理由（推論）**: **SG 側で明示的に制御**したほうが意図が分かりやすいためです。 ([AWS ドキュメント][7])

### `sslPolicy`

* **型**: `elbv2.SslPolicy`
* **必須か**: 任意
* **何に使う props か**: TLS ポリシーを決める
* **よく使う値（推論）**: `elbv2.SslPolicy.RECOMMENDED_TLS`
* **その値にする理由（推論）**: 既定任せより、TLS 方針をコードで見えるようにしたいためです。 ([AWS ドキュメント][7])

# 6. ALB Target Group 🎯

## 6-1. `elbv2.ApplicationTargetGroup`

`ApplicationTargetGroupProps` では `targetType`、`targets`、`vpc`、`port`、`protocol`、`healthCheck`、`deregistrationDelay`、`stickinessCookieDuration` などが主要です。`targetType` は自動判定、`healthCheck` の既定は target に依存、`deregistrationDelay` の既定は **300 秒**です。 ([AWS ドキュメント][8])

### `targetType`

* **型**: `elbv2.TargetType`
* **必須か**: 任意
* **何に使う props か**: target を Instance / IP どちらで扱うか決める
* **よく使う値（推論）**: Windows EC2 なら `elbv2.TargetType.INSTANCE`
* **その値にする理由（推論）**: Replatform の EC2 配下なら instance target が自然だからです。 ([AWS ドキュメント][8])

### `targets`

* **型**: `elbv2.IApplicationLoadBalancerTarget[]`
* **必須か**: 任意
* **何に使う props か**: ぶら下げる target を決める
* **よく使う値（推論）**: `asg` または `instance`
* **その値にする理由（推論）**: EC2 単体か ASG かで素直に切り替えられるためです。 ([AWS ドキュメント][8])

### `vpc`

* **型**: `ec2.IVpc`
* **必須か**: 任意
* **何に使う props か**: IP / InstanceId target の属する VPC を決める
* **よく使う値（推論）**: `vpc`
* **その値にする理由（推論）**: target のネットワーク所属を明示しておくためです。 ([AWS ドキュメント][8])

### `port`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: target 側受けポートを決める
* **よく使う値（推論）**: `80` / `8080` / `443`
* **その値にする理由（推論）**: ALB とアプリの責務分離に合わせてポートを固定したいからです。 ([AWS ドキュメント][8])

### `protocol`

* **型**: `elbv2.ApplicationProtocol`
* **必須か**: 任意
* **何に使う props か**: target との通信プロトコルを決める
* **よく使う値（推論）**: `HTTP`
* **その値にする理由（推論）**: TLS 終端を ALB に置くなら、背後は HTTP でシンプルにしやすいためです。 ([AWS ドキュメント][8])

### `healthCheck`

* **型**: `elbv2.HealthCheck`
* **必須か**: 任意
* **何に使う props か**: target 健全性の判定方法を決める
* **よく使う値（推論）**: `path: '/health'`、`healthyHttpCodes: '200'`
* **その値にする理由（推論）**: Windows IIS / 業務 Web でも、**専用ヘルス URL** を切るほうが判定が安定するためです。 ([AWS ドキュメント][8])

### `deregistrationDelay`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: 切り離し前の待ち時間を決める
* **よく使う値（推論）**: `cdk.Duration.seconds(30)` 〜 `120`
* **その値にする理由（推論）**: 接続を即切らず、ローリング更新時の影響を減らしたいからです。 ([AWS ドキュメント][8])

### `stickinessCookieDuration`

* **型**: `cdk.Duration`
* **必須か**: 任意
* **何に使う props か**: sticky session を有効化する
* **よく使う値（推論）**: **原則未設定**
* **その値にする理由（推論）**: まずはステートレス化を優先し、必要が明確な時だけ有効にしたいからです。 ([AWS ドキュメント][8])

# 7. CloudWatch Alarm 🔔

## 7-1. `cloudwatch.Alarm`

`AlarmProps` では `metric`、`threshold`、`evaluationPeriods` が核で、`comparisonOperator`、`datapointsToAlarm`、`treatMissingData`、`actionsEnabled` などを持ちます。GCAS でも **定量的計測 → ダッシュボード → 必要に応じてアラーム連携** の流れが示されています。 ([AWS ドキュメント][9])

### `metric`

* **型**: `cloudwatch.IMetric`
* **必須か**: **必須**
* **何に使う props か**: 何を監視するか決める
* **よく使う値（推論）**:

  * `alb.metricHttpCodeTarget(...)`
  * `alb.metricTargetResponseTime(...)`
  * `ec2.metricStatusCheckFailed()`
  * `cwAgent` 由来の `MemoryUtilization`
* **その値にする理由（推論）**: GCAS 的にも、**CPU だけでなく UX / エラー / 可用性** に近い指標を優先したいからです。 ([AWS ドキュメント][9])

### `threshold`

* **型**: `number`
* **必須か**: **必須**
* **何に使う props か**: しきい値を決める
* **よく使う値（推論）**: 指標ごとに個別設定
* **その値にする理由（推論）**: CPU 何% のような横並びではなく、**サービス影響が出る値**に合わせるべきだからです。 ([AWS ドキュメント][9])

### `evaluationPeriods`

* **型**: `number`
* **必須か**: **必須**
* **何に使う props か**: 何期間連続で比較するか決める
* **よく使う値（推論）**: `2` 〜 `3`
* **その値にする理由（推論）**: 単発スパイクで鳴らしすぎず、異常の継続を見たいからです。 ([AWS ドキュメント][9])

### `comparisonOperator`

* **型**: `cloudwatch.ComparisonOperator`
* **必須か**: 任意
* **何に使う props か**: しきい値との比較方法を決める
* **よく使う値（推論）**: `GREATER_THAN_OR_EQUAL_TO_THRESHOLD`
* **その値にする理由（推論）**: レイテンシ・エラー率・使用率系は、上振れ検知が基本になりやすいためです。 ([AWS ドキュメント][9])

### `datapointsToAlarm`

* **型**: `number`
* **必須か**: 任意
* **何に使う props か**: M/N 判定の M を決める
* **よく使う値（推論）**: `2` of `3`
* **その値にする理由（推論）**: ノイズを抑えつつ、継続的異常は拾いやすいためです。 ([AWS ドキュメント][9])

### `treatMissingData`

* **型**: `cloudwatch.TreatMissingData`
* **必須か**: 任意
* **何に使う props か**: 欠損データをどう扱うか決める
* **よく使う値（推論）**:

  * 死活・疎通系: `BREACHING` か `NOT_BREACHING` を要件で選択
  * バッチ系: `NOT_BREACHING`
* **その値にする理由（推論）**: メトリクス欠損の意味が監視対象で違うため、**一律ではなく使い分ける**必要があるからです。 ([AWS ドキュメント][9])

### `actionsEnabled`

* **型**: `boolean`
* **必須か**: 任意
* **何に使う props か**: 通知 / 自動処理を有効にするか決める
* **よく使う値（推論）**: 本番 `true`
* **その値にする理由（推論）**: 作るだけで通知しない alarm は、運用上の実効性が低いためです。 ([AWS ドキュメント][9])
